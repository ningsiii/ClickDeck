# 批注式意图交互可研 (Task 51A)

## 1. 结论

批注式意图交互应该作为 ClickDeck 下一阶段的核心增量，但第一版必须保持轻。

推荐第一版定位：

```text
用户点击“添加意图”
→ 在页面中框选一个区域
→ 选择动作
→ 输入一句意图
→ ClickDeck 生成结构化 prompt
→ 用户把 prompt 交给 Codex/Claude/Cursor 等代码工具执行
```

ClickDeck 仍然是“定位 + 上下文 + 约束”的翻译官，不是页面布局执行器。

第一版应先做 prompt-only 交互，不直接修改 DOM，不写回源码，不承诺 AI 一轮改好。现有 50B-50G 已经提供了足够的底层材料：视觉单元、区域框、区域上下文、prompt builder、move 双区域表达、ElementLocator 辅助字段。51A 的重点不是再造底层模型，而是把这些能力组织成用户能理解的轻交互。

## 2. 当前代码地基

当前可复用能力：

- `src/content/intent-region.ts`: 已有 `IntentAction = "add" | "delete" | "replace" | "restyle" | "move"`，并能生成 `IntentRegion` 与 `IntentOperation`。
- `src/content/region-context.ts`: 已能对区域框生成框内候选、附近参照、空白判断和 confidence。
- `src/export/intent-prompt.ts`: 已能把 `IntentOperation + RegionContext` 生成结构化 prompt，并支持 `move` 的 source/target 双区域。
- `src/content/visual-units.ts`: 已提供细粒度视觉单元基础设施。
- `src/content/dom-utils.ts`: `ElementLocator` 已有背景图、语义角色、语义父级、兄弟摘要和 selector stability 辅助信息。

当前缺口：

- 没有面向用户的“添加意图”入口。
- 没有区域框交互 UI。
- 没有贴着 ClickDeck 主面板的意图草稿栏。
- 没有把现有 intent prompt builder 接入面板和剪贴板。
- 没有意图区分、预期管理和错误状态文案。

## 3. 必须回答的问题

### 3.1 第一版动作集合

第一版建议分成“可展示 MVP”和“已具备底层能力但 UI 后置”。

可展示 MVP：

- `add`: 在选区或空白区域附近新增内容。
- `delete`: 删除选区内内容，并允许局部闭合空位。
- `replace`: 替换选区内内容，用户原话必须保留。
- `restyle`: 调整选区风格，必须附带 style reference 和限制范围。

可带实验入口或第二阶段接入：

- `move`: 已有双区域 prompt 能力，但交互要用户选择 source A 和 target B，心智比单区域动作重。建议在 51C 或 51D 单独实现。

必须延后：

- resize / scale 结构调整。
- 全局重排。
- 自动美化整页。
- 图表重绘或图表数据理解。
- 图片生成或图片替换素材生成。
- 任意 CSS 输入。
- 直接应用 AI 结果到页面。
- 自由拖拽、吸附、图层、组合、对齐分布。

原因：

- `add/delete/replace/restyle` 能直接复用单区域 `IntentRegion` 和 `RegionContext`。
- `move` 虽然 prompt builder 已支持，但 UI 是双阶段状态机，应该单独实现和测试。
- resize 和全局重排会把 ClickDeck 拉向布局引擎，与父级边界冲突。

### 3.2 空白处添加如何锚定

空白处添加不应该伪造一个被修改元素。正确表达是：

```text
sourceContext.empty === true
→ 使用 region.anchor 表达当前 slide/section/container
→ 使用 nearby references 表达上下左右参照
→ 使用 viewportBox/documentBox/relativeBox 表达视觉位置
→ 要求 AI 在该区域附近新增内容，并匹配当前区域风格
```

这里的 `nearby references` 不是 AI 猜测，也不是纯 CSS 规则匹配。第一版只做轻量计算：基于视觉单元的矩形位置，找框选区域上、下、左、右最近的可见参照物，每个方向最多 1 个。

`nearby` 的作用是告诉 coding AI：“这个区域大概靠近谁、应该参考谁对齐或保持风格一致”。它不是被修改对象，也不能强行替代用户框选的主目标。第一版不做复杂语义分析、不做深层 CSS 推断、不做 AI 判断。

具体规则：

- PPT-like 页面优先锚定当前 slide。
- 长页面优先锚定最近 section/card/article/main。
- 如果区域为空但 nearby 明确，confidence 为 medium，prompt 可以继续生成。
- 如果区域为空且 nearby 也为空，prompt 仍可生成，但 UI 应用温和提示说明“参照较少”，而不是禁止生成或吓阻用户。
- 不要强行把附近元素当成被修改对象；附近元素只是参照，不是主目标。

示例文案：

```text
这个区域周围参照较少。ClickDeck 会按当前页位置和整体风格生成说明；如果你希望更精确，可以框选靠近标题、图片或卡片的位置。
```

### 3.3 具体文案和抽象意图是否区分

需要让 coding AI 理解“具体文案”和“抽象意图”的差异，但第一版不应让用户学习开关，也不应在 ClickDeck 本地做规则匹配。

推荐第一版只用一个输入框：

- 输入框 label: `写一句你希望 AI 怎么改`
- placeholder: `例如：加一段关于价格优势的说明 / 把这里改得更科技感 / 替换成下面这句文案`

原则：

- ClickDeck 必须原样保留 `userIntent`，不要替用户改写、总结或分类。
- 不新增 `UserIntentMode` 开关。
- 不用“如果用户写了某个词就判定为具体文案”的本地规则匹配。
- prompt 中加入通用原则：如果用户意图足够明确，coding AI 直接执行；如果表述可能有多种理解，coding AI 应先追问用户，不要擅自扩大修改范围。

推荐写入所有 intent prompt 的通用原则：

```text
Global rules:
1. Use the original HTML structure as the source of truth.
2. Preserve the user's wording and intent.
3. Keep changes limited to the selected region and directly related surrounding layout.
4. Match the existing visual style unless the user explicitly asks for another style.
5. If the intent or placement is ambiguous, ask the user a clarifying question before editing.
```

空白区域只补充一条特有说明，不重复通用原则：

```text
The selected region is an empty visual area. Treat it as the intended placement area, not as an existing element to edit.
```

推荐 prompt 输出结构：

```text
ClickDeck edit instruction

Page:
- URL:
- Title:
- Scope: Current active browser page only.

Global rules:
1. ...

Operations:
1. Action:
   User intent:
   Target region:
   - Page mode:
   - Anchor:
   - Box:
   Region contents:
   - ...
   Nearby references:
   - above:
   Style reference:
   - ...
   To do:
   - ...
   Do not:
   - ...
```

`If the intent or placement is ambiguous...` 只放在 `Global rules`，不要在每个 operation 里重复堆。

### 3.4 是否需要新增 IntentPatch

第一版不建议新增 `IntentPatch` 或 `StructureIntentPatch`。

推荐策略：

- 第一版只做 prompt-only 记录。
- 意图操作不进入现有 undo/redo。
- 不进入保存恢复。
- 不影响 HTML/长图/图片 PDF/演示导出。
- 生成 prompt 后可以在弹窗里预览和复制。
- 区域框、意图输入卡片和 prompt preview 必须支持取消、关闭、删除和重新框选；这些属于临时 UI 状态，不是编辑历史。

原因：

- 现有 undo/redo 面向真实 DOM/style patch。
- 意图 prompt 不会改变页面，进入 undo/redo 反而制造错觉。
- 如果保存意图历史，需要新的“意图草稿/批注记录”模型，不应复用样式 patch。

后续如果需要持久化，建议新增独立模型：

```ts
type IntentAnnotation = {
  id: string;
  operation: IntentOperation;
  sourceContextSnapshot: RegionContextSnapshot;
  targetContextSnapshot?: RegionContextSnapshot;
  prompt: string;
  status: "draft" | "copied" | "discarded";
  createdAt: number;
};
```

注意：`RegionContext` 当前含 DOM element 引用，不能直接持久化。若要保存，必须设计可序列化 snapshot。

### 3.5 如何管理用户预期

产品文案必须明确：ClickDeck 是翻译官，不是执行者。这里不是新增一套复制系统，而是复用现有 `showPromptPreview()` 和 clipboard flow。

建议在 prompt preview 顶部显示短提示：

```text
这是给 AI/代码工具的修改说明。ClickDeck 只负责定位、上下文和约束，不会直接改源码，也不保证 AI 一轮完成。
```

按钮文案避免：

- `自动修改`
- `AI 美化`
- `应用修改`
- `一键生成`

推荐文案：

- `生成修改说明`
- `复制给 AI`
- `结构化 prompt`
- `意图说明`
- `添加意图`

错误状态文案：

- 参照较少: `这个区域周围参照较少。ClickDeck 会按当前位置和整体风格生成说明。`
- 框选区域太小: `框选区域太小，请重新框选需要说明的位置。`
- move 缺少 target: `移动需要再选择一个目标位置。`

### 3.6 如何隔离 undo/redo、保存恢复、导出链路

隔离原则：

- 意图交互只生成 prompt，不修改页面 DOM。
- 不调用 `applyStyleAction()`。
- 不写入 `EditorState.patches`。
- 不触发 undo/redo 可用状态。
- 不影响 saved edits notice。
- 不影响 `exportHtmlSnapshot()`、长图、图片 PDF、演示模式。

UI 层建议：

- 意图入口放在独立区块，例如 `添加意图` / `修改说明`。
- 不放进 Typography、Image、Background 这些确定性样式区域。
- Prompt preview 可复用现有 `showPromptPreview()`，但标题和说明要区分“样式改动 prompt”和“意图 prompt”。
- 意图草稿 UI 不贴在被框选区域旁边，而是贴在 ClickDeck 主面板旁边，作为临时意图草稿栏。

工程层建议：

- 新增独立 controller，例如 `intent-controller.ts` 或在 controller 内保持清晰函数边界。
- 复用 `buildIntentPrompt()`，不要改 `buildAiEditPrompt()`。
- 复用现有 clipboard flow，不新增外部 AI 调用。

## 4. 第一版交互流程

### 4.1 单区域动作

流程：

```text
用户点击 ClickDeck 面板里的“添加意图”
→ 页面进入区域框模式
→ 用户框选一个区域
→ ClickDeck 主面板旁边出现一张编辑态意图卡片
→ 用户选择动作：添加、删除、替换、调风格
→ 用户输入一句意图（delete 可选）
→ 用户点击“保存”或“取消”
→ 保存后的卡片进入折叠态，竖向排列在主面板旁边
→ 用户点击卡片可回到原框选区域并临时高亮该区域
→ 用户点击“生成修改说明”
→ ClickDeck 构建 RegionContext
→ buildIntentPrompt()
→ 展示 prompt preview
→ 用户复制给 AI
```

第一版意图交互只使用框选区域作为输入，不接入现有单元素选择。原因是现有单元素选择已经服务于确定性样式编辑，而意图交互表达的是“这里怎么改”，区域框更符合用户心智。

意图草稿栏规则：

- 草稿栏贴着 ClickDeck 主面板，不散落在页面各处。
- 第一版至少支持一条草稿；如实现多条，必须竖向排列。
- 每条草稿卡片包含动作、用户意图摘要、区域序号、删除按钮。
- 编辑态卡片包含动作选择、输入框、保存、取消。
- 保存后的卡片可折叠/展开。
- 点击保存后的卡片时，页面滚动到对应区域，并临时显示该区域框。
- 删除卡片只删除临时意图草稿，不影响页面 DOM、undo/redo、导出和已存在样式修改。

### 4.2 空白添加

流程：

```text
用户点击“框选意图区域”
→ 在空白处画框
→ 选择“添加”
→ 输入“加一段关于价格优势的说明”
→ ClickDeck 检测 empty + nearby
→ prompt 写明：这是空白视觉区域，作为新增或调整内容的位置，不是已有元素
```

如果 empty 且 nearby 为空，UI 不应禁止生成。只显示“参照较少”的轻提示，并让 prompt 使用当前 slide/section/container、相对坐标和整体风格摘要作为锚点。

### 4.3 移动到

建议后置为单独任务。

流程：

```text
用户框选源区域 A
→ 选择“移动到”
→ 页面提示“请选择目标位置”
→ 用户框选目标区域 B
→ 输入可选补充意图
→ 生成 move prompt
```

move 的 UI 必须清楚区分：

- Source region A: 被移动的内容。
- Target region B: 希望移动到的位置。

## 5. 建议拆出的 MVP 任务

目标：

- 收束 `buildIntentPrompt()`，让它输出弱模型友好的通用原则。
- 原样保留用户输入，不新增 `UserIntentMode`。
- 删除或避免“具体文案 vs 抽象意图”的本地规则匹配。
- 区分空白区域特有说明和通用原则，避免重复堆 prompt。
- 将 prompt 整理成 `Page`、`Global rules`、`Operations`、`To do`、`Do not` 的施工单结构。

不允许：

- 不做面板 UI。

- 不做真实 DOM 修改。
- 不做 move UI。
- 不进入 undo/redo。

验收：

## 6. 风险与约束

### 6.1 交互风险

- 区域框会让用户以为可以直接拖拽排版。
- 需要文案和 UI 状态明确“这是生成修改说明，不是直接改页面”。
- move 的双区域流程容易混淆，建议后置。

### 6.2 prompt 风险

- 弱模型可能把局部修改理解成全页重写。
- 必须保留明确的 `To do`、`Do not` 和 `Global rules`，避免弱模型把局部修改理解成全页重写。
- 用户原话必须原样保留，不要替用户润色。

### 6.3 数据风险

- `RegionContext` 含 DOM element，不能直接持久化。
- 如果后续做历史，需要 snapshot 化。
- 空白区域置信度低时，prompt 要诚实表达不确定性。

### 6.4 产品风险

- 这个能力容易被误解为“AI 自动改网页”。
- 第一版必须只做复制 prompt，不做自动执行。
- 不要把它包装成付费版承诺；先验证真实使用闭环。

## 7. 推荐下一步

推荐下一步执行：

```text
任务 51B：批注式意图数据与 prompt 接入 MVP
```

原因：

- 51B 是最小工程桥接任务，不碰 UI，风险低。
- 它能先把 50B-50G 的底层能力整理成稳定 API。
- 51C 再做 UI 时，小模型会有明确的数据入口，不需要边写 UI 边猜模型。

不建议直接执行 51C。当前 UI 复杂度已经较高，如果不先收束 intent action 和 prompt options，小模型容易把区域框、面板、prompt、undo/redo 混在一起。

## 8. 意图草稿持久化可研 (Task 51F)

### 8.1 序列化结构设计
如果强行进行持久化，我们需要解决 `RegionContext` 中的 DOM 引用问题。现有对象包含了指向具体页面节点的引用（如 `HTMLElement`），这在刷新页面后必然丢失，因此需要构造一层 `Snapshot`：

```ts
type VisualUnitSnapshot = Omit<VisualUnit, "element">;

type RegionContextSnapshot = Omit<RegionContext, "candidates" | "nearby"> & {
  candidates: {
    unit: VisualUnitSnapshot;
    rank: number;
    reason: string;
    overlapRatio: number;
    centerInBox: boolean;
  }[];
  nearby: {
    direction: "above" | "below" | "left" | "right";
    unit: VisualUnitSnapshot;
    distance: number;
    summary: string;
  }[];
};

export type IntentAnnotation = {
  id: string;
  operation: IntentOperation;
  sourceContextSnapshot: RegionContextSnapshot;
  targetContextSnapshot?: RegionContextSnapshot;
  prompt: string;
  status: "draft" | "copied" | "discarded";
  createdAt: number;
};
```
通过上述快照剔除 `element`，并保留基于 `ElementLocator`（支持按层级和属性重新查找）的锚点机制，理论上完全可以支撑草稿在 `localStorage` 级别的持久化落盘。

### 8.2 数据边界的隔离性
意图持久化数据必须严格与现有基建切割开来：
- **与 Style Patch (样式补丁) 的边界**：Style Patch 代表了已经执行的、确定性的视觉变动，应用加载时必须注入。而意图草稿本质是“修改需求”，严禁混入 Patch 流进行还原操作。
- **与 Undo/Redo 的边界**：不进入现有的 `History` 栈。创建草稿本身不改变页面，所以不支持撤销；编辑草稿仅在卡片内发生。
- **与导出的边界**：HTML Snapshot、长图下载以及 PPT 演示模式均要求“纯净的界面效果”。这些临时草稿 UI 组件（含双区的高亮框）必须在导出前进行全量卸载过滤。

### 8.3 最终结论：是否值得实现持久化？

**强烈建议：现阶段不实现持久化。**

**底层逻辑说明：**
ClickDeck 的核心心智并非 Figma 这种独立画布，它是基于现有正在工作的 DOM 上的“修改指示器”。
其理想的用户流是：
1. 产生想法 → 画框圈选 → 生成 Prompt。
2. 将 Prompt 送往诸如 Cursor / 外部大模型引擎，利用 AI 直接改写背后真实的 HTML/TS 源文件。
3. 保存源码，触发前端工具链（如 Vite 热更新），或者用户按 F5 刷新页面，以便校验修改。

在上述闭环中，一旦热更新或刷新发生，**意图就已经变成现实（DOM 改变了）**。
如果我们在 `localStorage` 挂载了意图草稿，当用户刷新后，他们会惊恐地发现：“页面上已经按要求多了一个按钮，但 ClickDeck 的草稿栏里，居然还有一个要求增加按钮的蓝框浮在那里！” 
这种“过期残留”不仅带来了无效的视觉垃圾，更由于底层 DOM 树的挪移，极易引发旧 Locator 查找错乱而导致蓝框飞出界面。

**总结**：意图草稿本质上是“一次性指令集 (Ephemeral Commands)”。在当前未打通与外部 AI 引擎实时通讯之前，把它作为纯**驻留内存的临时层**反而体验更清爽。用完即拷，刷新即走，避免陷入过重且极易陈旧的状态管理泥潭。
