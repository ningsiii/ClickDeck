  - 点击 `A+` 后字号变大。

## 1. 总体目标

v0.1 MVP 的目标是：

```text
用户打开一个 HTML 页面
  -> 启用 ClickDeck 插件
  -> 点选页面元素
  -> 用语义化控件微调视觉样式
  -> 可撤销/重做
  -> 可复制诊断信息
  -> 导出 HTML 快照
  -> 导出 PDF
```

非目标：

- 不接入 AI。
- 不生成内容。
- 不回写源码。
- 不做 Figma 式自由拖拽。
- 不做可编辑 PPT 导出。

## 5. 任务 2：建立编辑状态层

### 目标

当前代码直接改 DOM，没有稳定状态模型。后续撤销、导出、诊断都需要状态层。

### 需要建立的数据结构

建议先从简单结构开始：

```ts
type SelectedElementState = {
  element: HTMLElement;
  descriptor: string;
};

type StylePatch = {
  id: string;
  targetDescriptor: string;
  property: "fontSize" | "textAlign" | "color" | "margin" | "padding";
  before: string;
  after: string;
  createdAt: number;
};

type EditorState = {
  active: boolean;
  selected: SelectedElementState | null;
  patches: StylePatch[];
};
```

### 注意点

- 不要一开始追求完美 selector。
- `HTMLElement` 可以保存在运行时状态里，但导出/诊断要用可序列化 descriptor。
- descriptor 先用 `tag#id.class1.class2` 即可，后续再增强。

### 验收标准

- 选中元素会进入 state。
- 每次样式修改都会记录 patch。
- 当前行为不退化。
- E2E 仍通过。

## 6. 任务 3：撤销/重做

### 目标

任何会修改页面样式的动作都必须可撤销，这是用户安全感的核心。

### 功能范围

MVP 先支持：

- Undo：撤销上一次样式修改。
- Redo：恢复刚撤销的样式修改。
- 快捷键：
  - Windows/Linux: `Ctrl+Z`, `Ctrl+Shift+Z`
  - macOS 后续再补 `Meta+Z`

### 实现方向

每个 `StylePatch` 记录：

- 目标元素引用。
- 目标元素 descriptor。
- property。
- before。
- after。

Undo 时把 `before` 写回。

Redo 时把 `after` 写回。

### UI

面板上增加：

- Undo 按钮。
- Redo 按钮。

按钮在不可用时 disabled。

### 验收标准

- 点击 `A+` 后字号变大。
- Undo 后字号恢复。
- Redo 后字号再次变大。
- E2E 增加对应验证。

## 7. 任务 4：诊断日志缓冲

### 目标

现在 logger 只写 console。需要建立内存日志缓冲，后续用于“复制诊断信息”。

### 日志结构

```ts
type LogEntry = {
  id: string;
  level: "debug" | "info" | "warn" | "error";
  context: "extension" | "selection" | "style" | "export" | "diagnostics";
  message: string;
  details?: unknown;
  createdAt: number;
};
```

### 功能

- 保留最近 100 条日志。
- logger 仍然写 console。
- logger 同时写入内存 buffer。
- 提供 `getRecentLogs()`。
- 提供 `clearLogs()`。

### 后续用途

- UI 上增加“Copy diagnostics”。
- E2E 失败时可以快速知道哪一步失败。
- 用户提 issue 时可以复制诊断。

- logger 单元测试覆盖 buffer 写入。
- 不影响现有日志输出。

// --- Drag logic start ---
const header = element.querySelector<HTMLElement>(".clickdeck-panel__header");
if (header) {
  let isDragging = false;
  let offsetX = 0;
  let offsetY = 0;

  header.addEventListener("mousedown", (e: MouseEvent) => {
    // Ignore if clicking a button inside the header
    if ((e.target as HTMLElement).closest("button")) {
      return;
    }
    isDragging = true;
    offsetX = e.clientX - element.getBoundingClientRect().left;
    offsetY = e.clientY - element.getBoundingClientRect().top;
    e.preventDefault();
  });

  window.addEventListener("mousemove", (e: MouseEvent) => {
    if (!isDragging) {
      return;
    }
    const x = e.clientX - offsetX;
    const y = e.clientY - offsetY;
    element.style.left = `${x}px`;
    element.style.top = `${y}px`;
    element.style.right = "auto";
  });

  window.addEventListener("mouseup", () => {
    isDragging = false;
  });
}
// --- Drag logic end ---

case "pick-bg-color": {

  let bg = "";
  let current: HTMLElement | null = element.parentElement;
  while (current) {
    const computedBg = window.getComputedStyle(current).backgroundColor;
    // "transparent" or "rgba(0, 0, 0, 0)" are both considered transparent
    if (computedBg && computedBg !== "transparent" && computedBg !== "rgba(0, 0, 0, 0)") {
      bg = computedBg;
      break;
    }
    current = current.parentElement;
  }
  if (!bg) {
    logger.info("No non-transparent background found in ancestors");
    return null;
  }
  change = {
    property: "color",
    before: element.style.color,
    after: bg
  };
  break;
}

## 21. 下一阶段产品原则：AI 生成 HTML 的收尾台

### 定位订正

ClickDeck 下一阶段不应被描述为“轻量级编辑器”或“PPT 式 HTML 编辑器”，而应定位为：

```text
AI 生成 HTML 文档的浏览器内收尾工具。
```

英文定位建议：

```text
ClickDeck is a browser-based finishing tool for AI-generated HTML documents.
```

### 核心原则

1. **先修正，后创作**：ClickDeck 处理的是已经生成/已经渲染出来的 HTML 页面，不负责从零生成内容。
2. **先局部，后全局**：优先解决选中元素的文本、样式、图片和导出问题，不做整页重排。
3. **保留结构健康**：不得为了视觉方便默认引入自由拖拽、绝对定位、图层画板或任意对象插入。
4. **服务 AI 工作流**：人工视觉修改不仅可以导出，也可以被整理成 AI 能理解的修改意图，供 Cursor/Claude/v0/Bolt 等上游工具继续修改源码。
5. **不接入模型也能有 AI 价值**：第一版“AI 修改意图”只做本地差异总结，不调用 LLM，不需要 API key。

### 明确不做

- 不做 Figma/PPT 式自由画布。
- 不做任意拖拽移动。
- 不做自由角点缩放。
- 不做表格结构编辑。
- 不做画箭头、矩形、文本框等对象插入。
- 不做源码回写。
- 不接入外部 AI 模型。

## 22. 任务 19：稳定元素定位与修改意图目标描述

### 目标

为后续 `Copy AI edit prompt` 和图片/样式修改打基础：不要只用标题文字或标签名描述目标元素，而是生成一组稳定、可读、可回传给 AI 的目标定位信息。

### 为什么要做

仅靠 `h1` 或标题文本定位元素是不可靠的：

- 页面中可能有多个相同标题。
- 图片、图标、空容器没有文本。
- AI 生成页面经常有深层嵌套 `div`。
- 用户修改的是“某个卡片里的第二段文字”，而不是全局唯一的文本。
- 未来生成 AI prompt 时，需要同时给出人可读描述和机器可定位线索。

### 允许修改文件

- `src/content/dom-utils.ts`
- `src/content/dom-utils.test.ts`（如不存在则新增）
- `src/state/editor-state.ts`（仅允许新增定位字段类型）
- `src/content/controller.ts`（仅允许接入新的定位描述，不改交互逻辑）

### 不允许修改

- 不改 UI 面板布局。
- 不新增功能按钮。
- 不改导出逻辑。
- 不做图片替换。
- 不做 prompt 生成。
- 不引入新依赖。

### 实现要求

新增一个元素定位结构，例如：

```ts
type ElementLocator = {
  descriptor: string;
  tagName: string;
  roleHint?: string;
  textSnippet?: string;
  imageHint?: string;
  classHint?: string;
  idHint?: string;
  cssPath: string;
  nthOfTypePath: string;
  siblingIndex: number;
  parentDescriptor?: string;
};
```

推荐新增函数名：

```ts
export function createElementLocator(element: HTMLElement): ElementLocator
```

执行限制：

- 不删除现有 `describeElement`。
- 不删除或重命名现有 `targetDescriptor` 字段。
- 可以在 `StylePatch` / `ContentPatch` 中新增可选字段 `targetLocator?: ElementLocator`。
- `panel.setHint(...)` 第一版仍可继续使用 `locator.descriptor` 或现有 `describeElement(...)` 的输出。
- 如果某个字段暂时取不到，允许留空，但 `descriptor`、`tagName`、`cssPath`、`nthOfTypePath` 必须始终存在。

定位信息必须至少包含：

1. **人类可读描述**：例如 `h1 "Make the final page feel finished."`。

5. **CSS path**：生成从 `body` 到目标元素的 selector，优先用 `id`，否则用 tag + nth-of-type。
6. **父级描述**：记录最近一个有文本、id、class 或语义标签的父元素。

### 验收标准

- 对普通标题、段落、按钮、图片、无文本 `div` 都能生成定位信息。
- 重复标题场景下，`cssPath` 或 `nthOfTypePath` 能区分目标元素。
- 图片元素不会输出空洞的 `img`，必须包含 alt/src 文件名等辅助信息。
- 原有面板 hint 仍显示正常。
- 不改变当前选择/编辑/样式修改行为。

## 30. 任务 27：有效修改状态与撤销/保存一致性

### 目标

修正当前 `undo/redo`、本地保存和 AI prompt 之间的状态一致性：用户撤销后的修改不应继续出现在 AI prompt 或刷新恢复结果里。

### 为什么要做

当前实现已经有 `undo/redo`，也有 `state.patches`、`history.undoStack`、`chrome.storage.local` 和 AI prompt 生成。但真实用户会自然期待：

- 我撤销了，就代表这一步不再是当前结果。
- 我复制 AI prompt，应该描述当前页面最终状态，而不是历史上发生过的一切。
- 我刷新恢复，应该恢复当前有效修改，而不是恢复已经撤销的修改。

如果这一步不修，后续 prompt 预览做得再漂亮，也可能复制出不可信的内容。

### 允许修改文件

- `src/state/editor-state.ts`
- `src/state/history.ts`
- `src/state/editor-state.test.ts`
- `src/content/controller.ts`
- `src/export/change-summary.ts`
- `src/export/change-summary.test.ts`
- 相关测试文件

### 不允许修改

- 不重做整个 history 系统。
- 不新增复杂历史面板。
- 不新增依赖。
- 不改图片替换 UI。
- 不改导出 HTML/PDF 流程。
- 不做 AI 调用。

### 实现要求

1. 明确“当前有效修改”的来源。
   - 推荐以 `history.undoStack` 作为当前有效修改列表。
   - `state.patches` 可以继续保留为操作流水，但 AI prompt 和持久化必须只基于当前有效修改。
2. `undoLastPatch()` 后必须重新持久化当前有效修改。
3. `redoLastPatch()` 后必须重新持久化当前有效修改。
4. AI prompt 生成必须只读取当前有效修改，不能包含已撤销的 patch。
5. 页面恢复后，恢复出来的 patch 应进入当前有效修改栈，保证后续 undo 可以撤回恢复结果。
6. 如果继续保留 `state.patches`，必须避免它误导 prompt/storage；可以新增函数，例如：

```ts
function getEffectivePatches(): EditorPatch[] {
  return [...history.undoStack];
}
```

7. 如果用户撤销到没有任何有效修改，保存记录应清空或保存为空列表；刷新后不应再提示恢复无效修改。

### 验收标准

- 修改一个元素后复制 prompt，prompt 包含该修改。
- 撤销该修改后复制 prompt，不再包含该修改。
- 撤销后刷新页面，不会恢复已撤销的修改。
- redo 后复制 prompt，prompt 再次包含该修改。
- 保存/恢复、undo/redo、AI prompt 三者对“当前有效修改”的理解一致。

2. 单元测试：撤销后 prompt 不包含已撤销 patch。
3. 单元测试：撤销后序列化保存为空或不包含已撤销 patch。

5. E2E 可选：修改字号 -> undo -> 刷新 -> 启用 ClickDeck，不应恢复字号修改。

## 32. 任务 29：面板上下文化与完成区整理

### 目标

让面板从“所有按钮堆在一起”变成更符合人类操作的上下文工具面板：根据当前选中元素类型显示最相关的工具，并把导出/AI prompt 等完成动作集中到一个区域。

### 为什么要做

当前功能已经不少：文本、颜色、背景、圆角、间距、图片、导出、诊断、AI。真实用户会被过多按钮分散注意力。ClickDeck 的核心体验应该是：

```text
选中元素 -> 只看到此刻最相关的调整 -> 撤回/确认 -> 导出或复制 prompt
```

### 允许修改文件

- `src/content/panel.ts`
- `src/content/controller.ts`
- `src/content/i18n.ts`
- `src/content/overlay.ts`
- 相关测试文件

### 不允许修改

- 不重做整个 UI 视觉系统。
- 不引入 tab 组件库。
- 不新增依赖。
- 不删除现有功能。
- 不改变 style action 的语义。
- 不进入复杂 DOM 树面板。

### 实现要求

1. 面板需要知道当前选中元素类型：
   - `text`：普通文本/标题/段落等。
   - `image`：`img`。
   - `container`：常见容器，例如 `div/section/article/main/card-like`。
   - `none`：未选择元素。
2. 根据元素类型控制区域显隐或 disabled：
   - 未选择：禁用主要编辑按钮，只保留导出、诊断、AI 空状态入口。
   - 文本：显示字号、字重、行高、字间距、对齐、颜色、间距。
   - 图片：显示图片宽度、maxWidth、object-fit、图片圆角、替换图片、margin。
   - 容器：显示背景色、圆角、margin、padding、对齐（如适用）。
3. 图片工具不应在非图片元素上可点击。
4. 将导出与 AI prompt 归为一个“Finish/完成”区域：
   - Export HTML
   - Export PDF
   - Copy AI edit prompt
5. Diagnostics 保持在底部，不抢占主要编辑空间。
6. 保持面板可滚动，不能遮挡页面主要内容；不要扩大到半屏。
7. 面板 hint 应继续显示当前选中元素描述。

### 验收标准

- 未选中元素时，编辑按钮不会误作用于空目标。
- 选中文本时，图片替换和 object-fit 不可用或不可见。
- 选中图片时，图片相关按钮可用，且不会进入 `contenteditable`。
- 导出和 AI prompt 入口更容易找到。
- 原有样式修改、undo/redo、导出功能不回退。

### 建议测试

1. 单元/DOM 测试：`setSelectionContext("image")` 后图片区域可用。
2. 单元/DOM 测试：`setSelectionContext("text")` 后图片替换按钮 disabled 或隐藏。
3. E2E：选中文本，确认图片替换不可用。
4. E2E：选中图片，确认图片工具可用。

### 验证命令

npm run build
npm run typecheck
npm test
npm run e2e

## 33. 任务 30：保存恢复可见化与清空入口

### 目标

把当前基于 `confirm(...)` 和 `window.__clickdeckClearSavedEdits()` 的保存恢复体验，升级为普通用户可见、可理解、可操作的面板提示。

### 为什么要做

当前保存/恢复能力已经存在，但交互偏开发者：

- `confirm(...)` 会突然打断用户。
- `window.__clickdeckClearSavedEdits()` 只有开发者知道。
- 用户不知道当前页面是否有保存记录。
- 用户不知道恢复或清空是否成功。

这会影响信任感。

### 允许修改文件

- `src/content/controller.ts`
- `src/content/panel.ts`
- `src/content/i18n.ts`
- `src/content/overlay.ts`
- `src/state/editor-state.ts`（仅在需要补状态类型时）
- 相关测试文件

### 不允许修改

- 不做云同步。
- 不做多页面历史列表。
- 不做完整历史面板。
- 不保存完整 HTML。
- 不新增依赖。
- 不改变 storage key 规则，除非任务 27 发现必须修正。

### 实现要求

1. 启用 ClickDeck 时，如果当前 URL 有保存记录，不再立刻弹 `confirm(...)`。
2. 面板内显示极简恢复提示，例如：

```text
Saved edits found
[Restore] [Dismiss] [Clear]
```

3. 点击 `Restore`：
   - 恢复可定位 patch。
   - 跳过找不到目标的 patch。
   - 显示恢复结果，例如 `Restored 5 edits, skipped 1`。
4. 点击 `Dismiss`：
   - 仅隐藏提示。
   - 不删除保存记录。
5. 点击 `Clear`：
   - 清空当前页面保存记录。
   - 隐藏恢复提示。
   - 写日志。
6. 保留内部 `window.__clickdeckClearSavedEdits()` 可以，但不再作为唯一入口。
7. 恢复/清空失败必须给用户可见反馈并写日志。

### 验收标准

- 有保存记录时，面板显示可见恢复提示，而不是浏览器 confirm。
- 用户可以恢复、忽略或清空保存记录。
- 清空后刷新/重新启用不再提示恢复。
- 恢复失败或部分跳过不会崩溃。
- undo/redo 与恢复结果不冲突。

### 建议测试

1. 单元测试：有保存记录时 panel 显示 Restore/Dismiss/Clear。
2. 单元测试：Clear 调用 storage remove。
3. E2E：修改元素 -> 刷新 -> 启用 ClickDeck -> 点击 Restore -> 样式恢复。
4. E2E：修改元素 -> 刷新 -> 启用 ClickDeck -> 点击 Clear -> 再刷新不提示。

### 验证命令

```bash
npm run build
npm run typecheck
npm test
npm run e2e
```

```text
feat: show saved edits restore controls
```

## 34. 任务 31：关键人类流程 E2E 覆盖

### 目标

补齐 ClickDeck 当前最关键的人类使用链路 E2E 测试，避免后续低等级模型改 UI 或状态逻辑时破坏核心体验。

### 为什么要做

现在 E2E 主要覆盖“启用插件、选标题、字号变大、undo/redo”。这很重要，但还不够覆盖当前真实产品能力：

- Esc 取消选择。
- Tab/Shift+Tab 切换选择。
- 保存/恢复。
- 图片替换。
- AI prompt 生成。
- 撤销后 prompt 不应包含已撤销修改。

这些都是用户会实际走的路径。

### 允许修改文件

- `tests/e2e/clickdeck-extension.spec.ts`
- `fixtures/demo-page.html`
- `fixtures/*`（仅新增测试素材，例如小图片）
- 必要时小改测试 helper

### 不允许修改

- 不改产品功能代码，除非测试暴露真实 bug；如果需要修 bug，必须在执行记录里明确说明。
- 不新增大型测试框架。
- 不让测试依赖外网。
- 不测试截图像素细节。
- 不把测试写得过于脆弱，例如依赖按钮在面板中的绝对位置。

### 实现要求

至少新增以下 E2E：

1. **Esc 取消选择**
   - 启用 ClickDeck。
   - 选择文本元素。
   - 按 Esc。
   - 确认 selected outline 隐藏或 hint 回到选择提示。
   - 再点样式按钮不应修改原元素。

2. **Tab/Shift+Tab 切换选择**
   - 选择一个子元素。
   - 按 Tab。
   - 确认 hint 或 outline 对应父元素。
   - 按 Shift+Tab。
   - 确认回到子元素或第一个可选子元素。

3. **Prompt 生成**
   - 修改一个元素样式。
   - 打开 AI prompt。
   - 确认 prompt 包含 locator 和属性 before/after。

4. **撤销后 prompt 不包含撤销修改**
   - 修改一个元素。
   - undo。
   - 打开 AI prompt。
   - 确认不包含该已撤销修改。
   - 该用例依赖任务 27。

5. **图片替换与 undo/redo**
   - fixture 中提供图片。
   - 选中图片。
   - 通过 Playwright 上传本地小图片。
   - 确认 `img.src` 变成 data URL。
   - undo 后恢复旧 src。
   - redo 后恢复 data URL。

6. **保存/恢复**
   - 修改样式。
   - 刷新页面。
   - 启用 ClickDeck。
   - 恢复保存记录。
   - 确认样式恢复。
   - 该用例依赖任务 30。

### 验收标准

- E2E 不依赖外网。
- E2E 在本地 Edge/Chromium 插件模式稳定通过。
- 测试命名清楚，失败时能看出是哪条人类流程坏了。
- 原有 E2E 继续通过。

### 验证命令

```bash
npm run build
npm run typecheck
npm test
npm run e2e
```

```text

```
