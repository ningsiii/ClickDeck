# ClickDeck 执行路线图

## 0. 当前状态

当前项目已经完成：

- 产品方向确认：Chrome/Edge 当前页面可视化微调，不做 AI，不回写源码。
- MVP 范围确认：页面内编辑优先，HTML/PDF 导出后续进入；PPT 不进入 MVP。
- Git 仓库初始化，MIT License 已加入。
- Chrome MV3 + TypeScript + Vite 插件骨架已完成。
- Edge 真实浏览器 E2E 已跑通：
  - 加载 `dist/` 扩展。
  - 打开本地 demo 页面。
  - 通过 `Alt+Shift+C` 启用 ClickDeck。
  - 选择标题。
  - 点击 `A+` 后字号变大。

当前进度可理解为 v0.1 MVP 的约 15%。

## 1. 总体目标

v0.1 MVP 的目标是：

```text
用户打开一个 HTML 页面
  -> 启用 ClickDeck 插件
  -> 点选页面元素
  -> 用语义化控件微调视觉样式
  -> 可撤销/重做
  -> 可复制诊断信息
  -> 导出 HTML 快照
  -> 导出 PDF
```

非目标：

- 不接入 AI。
- 不生成内容。
- 不回写源码。
- 不做 Figma 式自由拖拽。
- 不做可编辑 PPT 导出。

## 4. 任务 1：拆分 content script

### 目标

当前 `src/content/main.ts` 已经超过 300 行，里面混合了：

- 激活/停用逻辑。
- DOM overlay。
- panel UI。
- 元素选择。
- 样式修改。
- 日志。

下一步应该先拆模块，避免后续继续堆大文件。

### 建议目录

```text
src/content/
  main.ts
  controller.ts
  overlay.ts
  panel.ts
  selection.ts
  style-actions.ts
  dom-utils.ts

src/state/
  editor-state.ts
  history.ts
  style-token.ts

src/shared/
  messages.ts
  types.ts
```

### 拆分边界

- `main.ts`：只负责接收 extension message 和初始化 controller。
- `controller.ts`：负责 activate/deactivate、事件绑定、模块协调。
- `overlay.ts`：负责描边框创建、定位、显示/隐藏。
- `panel.ts`：负责面板 DOM、按钮事件、选中元素描述显示。
- `selection.ts`：负责 hover/click 获取可编辑目标。
- `style-actions.ts`：负责把按钮动作应用到元素。
- `dom-utils.ts`：放 `describeElement`、`getEditableTarget` 等工具。

### 验收标准

- 行为不变。
- `Alt+Shift+C` 仍能启用/停用。
- E2E 仍通过。
- `main.ts` 明显变薄。

### 验证

```bash
npm run build
npm run typecheck
npm test
npm run e2e
```

### 建议 commit

```text
refactor: split content script modules
```

## 5. 任务 2：建立编辑状态层

### 目标

当前代码直接改 DOM，没有稳定状态模型。后续撤销、导出、诊断都需要状态层。

### 需要建立的数据结构

建议先从简单结构开始：

```ts
type SelectedElementState = {
  element: HTMLElement;
  descriptor: string;
};

type StylePatch = {
  id: string;
  targetDescriptor: string;
  property: "fontSize" | "textAlign" | "color" | "margin" | "padding";
  before: string;
  after: string;
  createdAt: number;
};

type EditorState = {
  active: boolean;
  selected: SelectedElementState | null;
  patches: StylePatch[];
};
```

### 注意点

- 不要一开始追求完美 selector。
- `HTMLElement` 可以保存在运行时状态里，但导出/诊断要用可序列化 descriptor。
- descriptor 先用 `tag#id.class1.class2` 即可，后续再增强。

### 验收标准

- 选中元素会进入 state。
- 每次样式修改都会记录 patch。
- 当前行为不退化。
- E2E 仍通过。

### 建议 commit

```text
feat: add editor state model
```

## 6. 任务 3：撤销/重做

### 目标

任何会修改页面样式的动作都必须可撤销，这是用户安全感的核心。

### 功能范围

MVP 先支持：

- Undo：撤销上一次样式修改。
- Redo：恢复刚撤销的样式修改。
- 快捷键：
  - Windows/Linux: `Ctrl+Z`, `Ctrl+Shift+Z`
  - macOS 后续再补 `Meta+Z`

### 实现方向

每个 `StylePatch` 记录：

- 目标元素引用。
- 目标元素 descriptor。
- property。
- before。
- after。

Undo 时把 `before` 写回。

Redo 时把 `after` 写回。

### UI

面板上增加：

- Undo 按钮。
- Redo 按钮。

按钮在不可用时 disabled。

### 验收标准

- 点击 `A+` 后字号变大。
- Undo 后字号恢复。
- Redo 后字号再次变大。
- E2E 增加对应验证。

### 建议 commit

```text
feat: add undo redo for style edits
```

## 7. 任务 4：诊断日志缓冲

### 目标

现在 logger 只写 console。需要建立内存日志缓冲，后续用于“复制诊断信息”。

### 日志结构

```ts
type LogEntry = {
  id: string;
  level: "debug" | "info" | "warn" | "error";
  context: "extension" | "selection" | "style" | "export" | "diagnostics";
  message: string;
  details?: unknown;
  createdAt: number;
};
```

### 功能

- 保留最近 100 条日志。
- logger 仍然写 console。
- logger 同时写入内存 buffer。
- 提供 `getRecentLogs()`。
- 提供 `clearLogs()`。

### 后续用途

- UI 上增加“Copy diagnostics”。
- E2E 失败时可以快速知道哪一步失败。
- 用户提 issue 时可以复制诊断。

### 验收标准

- logger 单元测试覆盖 buffer 写入。
- 不影响现有日志输出。
- build/typecheck/test/e2e 全通过。

### 建议 commit

```text
feat: add diagnostics log buffer
```

## 8. 任务 5：整理面板 UI

### 目标

当前 panel 是临时 HTML 字符串。需要变成更稳定、可扩展的 UI 模块。

### MVP 面板分区

建议分成：

- Header：ClickDeck、状态、关闭按钮。
- Selection：当前选中元素 descriptor。
- Typography：A-、A+。
- Alignment：Left、Center、Right。
- Color：Accent、Reset Color。
- History：Undo、Redo。
- Diagnostics：Copy diagnostics。

### 注意点

- 暂时不要引 React。
- Content script 内直接用 DOM API 足够。
- UI 结构要可测试，按钮要有清晰 accessible name。

### 验收标准

- E2E 仍能按按钮名称找到 `A+`。
- 面板不遮挡 selected outline 的逻辑不必现在完美，但不能挡住基础测试。
- 面板按钮状态与 history 状态同步。

### 建议 commit

```text
feat: organize editor panel controls
```

## 9. 任务 6：扩展样式能力

### 目标

从 PoC 的 `A+`、对齐、Accent 扩展到更像产品的语义化微调。

### 第一批能力

- 字号：小一点 / 大一点。
- 对齐：左 / 中 / 右。
- 颜色：强调色 / 恢复默认。
- 字重：弱化 / 正常 / 强调。
- 间距：紧凑 / 标准 / 宽松。

### 重要原则

- 先用 inline style 实现当前页面修改。
- 不回写源码。
- 不追求 Tailwind/class 智能替换。
- 每个动作必须记录 patch，支持 undo/redo。

### 验收标准

- 每个 style action 有单元测试或 E2E 覆盖。
- 每个 action 可撤销。
- 页面复杂元素不崩。

### 建议 commit

```text
feat: add semantic style actions
```

## 10. 任务 7：HTML 快照导出

### 目标

导出当前修改后的页面 HTML，让用户可以继续打开和展示。

### 实现方向

MVP 可以先：

- clone `document.documentElement`。
- 移除 ClickDeck 自身 UI：`[data-clickdeck='true']` 和 `#clickdeck-style`。
- 保留用户页面上已经写入的 inline style。
- 生成完整 HTML 字符串。
- 触发下载 `.html` 文件。

### 注意点

- 不要把 ClickDeck panel 导出进去。
- 不要上传用户页面。
- 不处理复杂外链资源内联化，先保留原 URL。

### 验收标准

- 修改标题字号后导出 HTML。
- 打开导出的 HTML，修改仍在。
- ClickDeck UI 不出现在导出文件里。

### 建议 commit

```text
feat: export edited html snapshot
```

## 11. 任务 8：PDF 长页面导出

### 目标

先实现最稳的长页面 PDF 导出模式。

### 可能路线

优先路线：

- 触发浏览器打印：`window.print()`。
- 注入 `@media print` 辅助样式。
- 隐藏 ClickDeck UI。

备选路线：

- 后续研究 Paged.js。
- 后续研究 html2pdf.js/html2canvas。

### 注意点

- 浏览器插件不能静默保存 PDF，用户仍可能看到打印对话框。
- MVP 可以接受“打开打印对话框，用户选择保存 PDF”。
- 关键是页面在打印时隐藏 ClickDeck UI。

### 验收标准

- 面板有 Export PDF 按钮。
- 点击后触发打印流程。
- 打印预览里没有 ClickDeck UI。
- 长页面内容可见。

### 建议 commit

```text
feat: add long page pdf export
```

## 12. 任务 9：PDF 16:9 和 A4 模式

### 目标

在长页面 PDF 跑通后，再加导出模式选择。

### 模式

- Long page：默认。
- Slides 16:9：适合 HTML slides。
- A4：适合文档。

### 实现方向

先用 print CSS：

```css
@page {
  size: A4;
  margin: 16mm;
}
```

16:9 可以先尝试：

- 识别 `section`、`.slide`、`.page`。
- 给这些容器设置 page break。
- 暂不追求所有页面完美。

### 验收标准

- UI 能选择模式。
- 选择 A4 后打印样式变为 A4。
- 选择 16:9 后 slide/section 尽量分页。

### 建议 commit

```text
feat: add pdf export modes
```
