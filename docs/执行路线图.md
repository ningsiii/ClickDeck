  - 点击 `A+` 后字号变大。

## 1. 总体目标

v0.1 MVP 的目标是：

```text
用户打开一个 HTML 页面
  -> 启用 ClickDeck 插件
  -> 点选页面元素
  -> 用语义化控件微调视觉样式
  -> 可撤销/重做
  -> 可复制诊断信息
  -> 导出 HTML 快照
  -> 导出 PDF
```

非目标：

- 不接入 AI。
- 不生成内容。
- 不回写源码。
- 不做 Figma 式自由拖拽。
- 不做可编辑 PPT 导出。

## 5. 任务 2：建立编辑状态层

### 目标

当前代码直接改 DOM，没有稳定状态模型。后续撤销、导出、诊断都需要状态层。

### 需要建立的数据结构

建议先从简单结构开始：

```ts
type SelectedElementState = {
  element: HTMLElement;
  descriptor: string;
};

type StylePatch = {
  id: string;
  targetDescriptor: string;
  property: "fontSize" | "textAlign" | "color" | "margin" | "padding";
  before: string;
  after: string;
  createdAt: number;
};

type EditorState = {
  active: boolean;
  selected: SelectedElementState | null;
  patches: StylePatch[];
};
```

### 注意点

- 不要一开始追求完美 selector。
- `HTMLElement` 可以保存在运行时状态里，但导出/诊断要用可序列化 descriptor。
- descriptor 先用 `tag#id.class1.class2` 即可，后续再增强。

### 验收标准

- 选中元素会进入 state。
- 每次样式修改都会记录 patch。
- 当前行为不退化。
- E2E 仍通过。

## 6. 任务 3：撤销/重做

### 目标

任何会修改页面样式的动作都必须可撤销，这是用户安全感的核心。

### 功能范围

MVP 先支持：

- Undo：撤销上一次样式修改。
- Redo：恢复刚撤销的样式修改。
- 快捷键：
  - Windows/Linux: `Ctrl+Z`, `Ctrl+Shift+Z`
  - macOS 后续再补 `Meta+Z`

### 实现方向

每个 `StylePatch` 记录：

- 目标元素引用。
- 目标元素 descriptor。
- property。
- before。
- after。

Undo 时把 `before` 写回。

Redo 时把 `after` 写回。

### UI

面板上增加：

- Undo 按钮。
- Redo 按钮。

按钮在不可用时 disabled。

### 验收标准

- 点击 `A+` 后字号变大。
- Undo 后字号恢复。
- Redo 后字号再次变大。
- E2E 增加对应验证。

## 7. 任务 4：诊断日志缓冲

### 目标

现在 logger 只写 console。需要建立内存日志缓冲，后续用于“复制诊断信息”。

### 日志结构

```ts
type LogEntry = {
  id: string;
  level: "debug" | "info" | "warn" | "error";
  context: "extension" | "selection" | "style" | "export" | "diagnostics";
  message: string;
  details?: unknown;
  createdAt: number;
};
```

### 功能

- 保留最近 100 条日志。
- logger 仍然写 console。
- logger 同时写入内存 buffer。
- 提供 `getRecentLogs()`。
- 提供 `clearLogs()`。

### 后续用途

- UI 上增加“Copy diagnostics”。
- E2E 失败时可以快速知道哪一步失败。
- 用户提 issue 时可以复制诊断。

- logger 单元测试覆盖 buffer 写入。
- 不影响现有日志输出。

```typescript
// --- Drag logic start ---
const header = element.querySelector<HTMLElement>(".clickdeck-panel__header");
if (header) {
  let isDragging = false;
  let offsetX = 0;
  let offsetY = 0;

  header.addEventListener("mousedown", (e: MouseEvent) => {
    // Ignore if clicking a button inside the header
    if ((e.target as HTMLElement).closest("button")) {
      return;
    }
    isDragging = true;
    offsetX = e.clientX - element.getBoundingClientRect().left;
    offsetY = e.clientY - element.getBoundingClientRect().top;
    e.preventDefault();
  });

  window.addEventListener("mousemove", (e: MouseEvent) => {
    if (!isDragging) {
      return;
    }
    const x = e.clientX - offsetX;
    const y = e.clientY - offsetY;
    element.style.left = `${x}px`;
    element.style.top = `${y}px`;
    element.style.right = "auto";
  });

  window.addEventListener("mouseup", () => {
    isDragging = false;
  });
}
// --- Drag logic end ---
```

```typescript
case "pick-bg-color": {

  let bg = "";
  let current: HTMLElement | null = element.parentElement;
  while (current) {
    const computedBg = window.getComputedStyle(current).backgroundColor;
    // "transparent" or "rgba(0, 0, 0, 0)" are both considered transparent
    if (computedBg && computedBg !== "transparent" && computedBg !== "rgba(0, 0, 0, 0)") {
      bg = computedBg;
      break;
    }
    current = current.parentElement;
  }
  if (!bg) {
    logger.info("No non-transparent background found in ancestors");
    return null;
  }
  change = {
    property: "color",
    before: element.style.color,
    after: bg
  };
  break;
}
```

## 21. 下一阶段产品原则：AI 生成 HTML 的收尾台

### 定位订正

ClickDeck 下一阶段不应被描述为“轻量级编辑器”或“PPT 式 HTML 编辑器”，而应定位为：

```text
AI 生成 HTML 文档的浏览器内收尾工具。
```

英文定位建议：

```text
ClickDeck is a browser-based finishing tool for AI-generated HTML documents.
```

### 核心原则

1. **先修正，后创作**：ClickDeck 处理的是已经生成/已经渲染出来的 HTML 页面，不负责从零生成内容。
2. **先局部，后全局**：优先解决选中元素的文本、样式、图片和导出问题，不做整页重排。
3. **保留结构健康**：不得为了视觉方便默认引入自由拖拽、绝对定位、图层画板或任意对象插入。
4. **服务 AI 工作流**：人工视觉修改不仅可以导出，也可以被整理成 AI 能理解的修改意图，供 Cursor/Claude/v0/Bolt 等上游工具继续修改源码。
5. **不接入模型也能有 AI 价值**：第一版“AI 修改意图”只做本地差异总结，不调用 LLM，不需要 API key。

### 明确不做

- 不做 Figma/PPT 式自由画布。
- 不做任意拖拽移动。
- 不做自由角点缩放。
- 不做表格结构编辑。
- 不做画箭头、矩形、文本框等对象插入。
- 不做源码回写。
- 不接入外部 AI 模型。

## 22. 任务 19：稳定元素定位与修改意图目标描述

### 目标

为后续 `Copy AI edit prompt` 和图片/样式修改打基础：不要只用标题文字或标签名描述目标元素，而是生成一组稳定、可读、可回传给 AI 的目标定位信息。

### 为什么要做

仅靠 `h1` 或标题文本定位元素是不可靠的：

- 页面中可能有多个相同标题。
- 图片、图标、空容器没有文本。
- AI 生成页面经常有深层嵌套 `div`。
- 用户修改的是“某个卡片里的第二段文字”，而不是全局唯一的文本。
- 未来生成 AI prompt 时，需要同时给出人可读描述和机器可定位线索。

### 允许修改文件

- `src/content/dom-utils.ts`
- `src/content/dom-utils.test.ts`（如不存在则新增）
- `src/state/editor-state.ts`（仅允许新增定位字段类型）
- `src/content/controller.ts`（仅允许接入新的定位描述，不改交互逻辑）

### 不允许修改

- 不改 UI 面板布局。
- 不新增功能按钮。
- 不改导出逻辑。
- 不做图片替换。
- 不做 prompt 生成。
- 不引入新依赖。

### 实现要求

新增一个元素定位结构，例如：

```ts
type ElementLocator = {
  descriptor: string;
  tagName: string;
  roleHint?: string;
  textSnippet?: string;
  imageHint?: string;
  classHint?: string;
  idHint?: string;
  cssPath: string;
  nthOfTypePath: string;
  siblingIndex: number;
  parentDescriptor?: string;
};
```

推荐新增函数名：

```ts
export function createElementLocator(element: HTMLElement): ElementLocator
```

执行限制：

- 不删除现有 `describeElement`。
- 不删除或重命名现有 `targetDescriptor` 字段。
- 可以在 `StylePatch` / `ContentPatch` 中新增可选字段 `targetLocator?: ElementLocator`。
- `panel.setHint(...)` 第一版仍可继续使用 `locator.descriptor` 或现有 `describeElement(...)` 的输出。
- 如果某个字段暂时取不到，允许留空，但 `descriptor`、`tagName`、`cssPath`、`nthOfTypePath` 必须始终存在。

定位信息必须至少包含：

1. **人类可读描述**：例如 `h1 "Make the final page feel finished."`。

5. **CSS path**：生成从 `body` 到目标元素的 selector，优先用 `id`，否则用 tag + nth-of-type。
6. **父级描述**：记录最近一个有文本、id、class 或语义标签的父元素。

### 验收标准

- 对普通标题、段落、按钮、图片、无文本 `div` 都能生成定位信息。
- 重复标题场景下，`cssPath` 或 `nthOfTypePath` 能区分目标元素。
- 图片元素不会输出空洞的 `img`，必须包含 alt/src 文件名等辅助信息。
- 原有面板 hint 仍显示正常。
- 不改变当前选择/编辑/样式修改行为。

## 30. 任务 27：有效修改状态与撤销/保存一致性

### 目标

- 我刷新恢复，应该恢复当前有效修改，而不是恢复已经撤销的修改。

## 36. 任务 33：Prompt 页面、Slide 定位与语言完善

### 目标

在已完成的 AI prompt 功能基础上补齐当前页面定位、HTML PPT/slide 定位和中英文 prompt 切换，让 prompt 更适合交给 Claude/Cursor/v0/Bolt 继续修改源 HTML/CSS。

### 为什么要做

ClickDeck 的 prompt 不是给用户看的日志，而是给 AI 执行的设计修改指令。它需要同时回答三件事：

- 这是哪个页面上的修改？
- 如果是 HTML PPT，这是哪一页/哪张 slide 上的修改？
- 用户希望用英文还是中文和 AI 沟通？

注意：这不是多网页编辑功能。ClickDeck 仍然只编辑当前浏览器 tab 的当前页面。

### 允许修改文件

- `src/export/change-summary.ts`
- `src/export/change-summary.test.ts`
- `src/content/controller.ts`
- `src/content/panel.ts`
- `src/content/i18n.ts`
- `src/content/dom-utils.ts`（仅允许新增/复用保守定位工具）
- 相关测试文件

### 不允许修改

- 不调用 AI。
- 不新增依赖。
- 不做多页面/多 tab 批量编辑。
- 不扫描同站点其它页面。
- 不把 ClickDeck 扩展成站点级编辑器。
- 不做复杂 slide deck runtime。
- 不导出 PPTX。

### 前置条件

- 任务 27-31 已完成。
- 如果当前代码已经有 prompt 预览 UI，本任务只在其上补页面/slide/语言，不要重做预览 UI。
- 如果当前代码尚未支持 prompt 预览，则先停止并记录阻塞，不要把本任务降级成“直接复制 prompt”。

### 推荐实现边界

1. prompt 生成函数显式接收上下文，不要在 `change-summary.ts` 内直接读 `window`：

```ts
type PromptLanguage = "en" | "zh";

type PromptPageContext = {
  url: string;
  title: string;
};

type PromptBuildOptions = {
  language: PromptLanguage;
  page: PromptPageContext;
};
```

2. `controller.ts` 负责收集：
   - `window.location.href`
   - `document.title`
   - 当前 prompt 语言
3. `change-summary.ts` 只负责把输入结构转成 prompt 字符串。
4. prompt 默认英文，预览 UI 中允许切换中文。
5. 用户手动编辑了 textarea 后，如果切换语言会覆盖内容，必须做一个简单确认，或只在未编辑状态下直接切换。

### Slide/Page 定位要求

对于 HTML PPT / slide deck 页面，要区分：

- **多个网页**：不支持，不做。
- **同一个 HTML 内的多页/多 slide**：可以在 prompt 里辅助定位。

第一版只做保守识别，不做完整 PPT 解析器：

1. 如果目标元素祖先命中以下结构，尝试生成 `Slide/Page`：
   - `section`
   - `.slide`
   - `.page`
   - `[data-slide]`
   - `[data-page]`
   - `[aria-roledescription="slide"]`
2. 优先取该祖先的：
   - `data-slide` / `data-page`
   - `aria-label`
   - 第一个标题文本（`h1/h2/h3`）
   - nth-of-type 或 css path
3. 识别不到就不写 `Slide/Page`，不要猜。
4. 每条修改仍必须包含原本的 locator，slide/page 只是辅助定位，不替代 locator。

### Prompt 内容要求

英文 prompt 开头必须包含：

```text
Page:
- URL: ...
- Title: ...
- Scope: Current active browser page only.
```

中文 prompt 开头必须包含：

```text
页面：
- URL：...
- 标题：...
- 范围：仅当前浏览器活动页面。
```

如果某条修改能识别 slide/page，则在该条修改里补充：

```text
Slide/Page: 3 "Market shift"
```

中文版本可写：

```text
页面/幻灯片：3「Market shift」
```

### 图片替换：AI Prompt 内容与用户提醒必须分层

图片替换时不要把完整 data URL/base64 写进 prompt，因为：

- base64 非常长，会污染 prompt。
- 很多代码项目不适合直接把图片内联进源码。
- AI 更需要知道“这里换成了用户提供的图片”，而不是读取一大段二进制文本。

必须区分两类内容：

1. **AI prompt 内容**：会被复制给 AI，必须是明确、可执行的修改指令。
2. **用户提醒内容**：只显示在 ClickDeck 预览 UI 里，用来提醒用户复制 prompt 后还需要附图或提供图片路径，不应混进主要 prompt 正文里当作产品说明。

英文 prompt 中图片替换条目建议写成类似：

```text
3. Target: img "Product screenshot"
   Locator: body > main:nth-of-type(1) > section:nth-of-type(2) > img:nth-of-type(1)
   Attribute: src should be replaced with the user-provided image asset.
   If no image file or asset path is provided with this prompt, ask the user for the replacement image before changing this src.
```

中文 prompt 中图片替换条目建议写成类似：

```text
3. 目标：img「Product screenshot」
   定位：body > main:nth-of-type(1) > section:nth-of-type(2) > img:nth-of-type(1)
   属性：src 需要替换为用户提供的图片素材。
   如果这份 prompt 没有同时提供图片文件或资源路径，请先向用户索要替换图片，再修改这个 src。
```

预览 UI 中必须把用户提醒放在 prompt 之外，例如在 textarea 上方或下方显示醒目的提示条：

```text
Image reminder: This prompt references a replaced image. When sending it to AI, attach the image file or provide the asset path. The full data URL is intentionally not copied.
```

中文：

```text
图片提醒：这份 prompt 提到了替换图片。发送给 AI 时，请额外附上图片文件，或提供项目中的图片路径。完整 data URL 已刻意不复制。
```

UI 提醒建议使用较醒目的样式，例如浅红/浅橙背景、图标或 `Warning` 文案，但不要打断复制流程。

第一版不需要做附件管理，也不需要上传图片；只需要：

- prompt 内告诉 AI：如果没收到图片/路径，要向用户索要。
- UI 内提醒用户：复制 prompt 后还要附图片或资源路径。
- 不复制完整 data URL/base64。

### 验收标准

- prompt 默认英文。
- 用户能在预览中切换中文 prompt。
- 英文和中文 prompt 都包含当前 URL 与 title。
- prompt 明确范围是当前活动页面，不暗示批量修改多个网页。
- 对 HTML PPT / slide deck，如能安全识别 slide/page，prompt 条目包含 slide/page 信息。
- 识别不到 slide/page 时不编造。
- 图片 data URL 仍不能完整出现在 prompt 中。

## 38. 任务 35：AI Prompt 按元素分组与最终变化合并

### 目标

优化 AI edit prompt 的结构：不要把每一个 patch 都机械输出成一条，而是按目标元素分组，把同一元素上的多项修改合并到同一个编号下，并对同一属性的连续修改只保留最终 before/after。

### 为什么要做

AI 更擅长执行结构清晰、条目明确的修改指令。当前 prompt 生成方式如果按 patch 顺序逐条输出，会出现：

- 同一个标题改了字号和颜色，却被拆成两条。
- 用户连续点三次 `A+`，prompt 里出现三条字号变化。
- AI 读到很多中间步骤，反而更容易误解最终目标。

更好的格式是：

```text
1. Target: h1 "如何优雅地改 HTML"
   Locator: body > main > section:nth-of-type(1) > h1
   Style: fontSize changed from "56px" to "64px".
   Style: color changed from "#222222" to "#1d4ed8".

2. Target: paragraph starting with "不用反复指挥 AI..."
   Locator: body > main > section:nth-of-type(1) > p
   Text changed from "..." to "...".
```

这样 AI 能逐条执行，也能知道每个元素最终要变成什么状态。

- `src/export/change-summary.test.ts`
- `src/state/editor-state.ts`（仅在需要补类型 helper 时）

### 推荐实现结构

为了降低误实现概率，建议拆出纯函数，不要把所有逻辑写在字符串拼接循环里：

```ts

  key: string;
  target: string;
  locator: string;
  slidePage?: string;
  styleChanges: Map<StyleProperty, { before: string; after: string }>;
  textChange?: { before: string; after: string };
  attributeChanges: Map<string, { before: string; after: string }>;
  firstSeenAt: number;
};

function groupPromptChanges(patches: EditorPatch[]): PromptChangeGroup[] {
  // pure grouping + merge logic
}
```

`buildAiEditPrompt(...)` 应只负责：

1. 调用分组/合并函数。
2. 过滤空 group。
3. 根据语言把 group 渲染成 prompt。

如果不采用这个函数名也可以，但必须有等价的纯函数或清晰分层，方便单测直接覆盖。

### 分组规则

1. 分组 key 必须稳定，推荐优先使用：
   - `targetLocator.cssPath`
   - fallback: `targetLocator.nthOfTypePath`
   - fallback: `targetDescriptor`
2. 同一分组内输出一次：
   - `Target`
   - `Locator`
   - 如已支持，输出 `Slide/Page`
3. 同一分组内可以包含多条变化：
   - 多个样式属性。
   - 文本变化。
   - 图片 `src` 属性变化。
4. 输出顺序应尽量保持用户修改的首次出现顺序，方便理解。

### 合并规则

1. 同一个元素 + 同一个 style property 连续或多次修改，只输出最终变化：

```text
fontSize: 16px -> 22px
```

不要输出：

```text
fontSize: 16px -> 18px
fontSize: 18px -> 20px
fontSize: 20px -> 22px
```

2. 合并时 `before` 应取该属性第一次修改前的值，`after` 应取最后一次修改后的值。
3. 如果最终 `after` 等于最初 `before`，说明用户改了又改回去了；该变化不应出现在 prompt 中。
4. content/text patch：
   - 同一元素多次文本修改，只输出第一次 before 到最后一次 after。
   - 如果最后文本等于最初文本，则不输出该文本变化。
5. attribute patch：
   - 同一元素同一 attribute 多次变化，只输出第一次 before 到最后一次 after。
   - 图片 data URL 仍必须隐藏为 `[data URL image]` 或中文等价占位。
6. 如果某个分组内所有变化都被合并抵消，则整个分组不输出。

英文：

```text

```

中文：

