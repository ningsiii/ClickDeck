# ClickDeck 工程执行原则与协作规则

## 1. 项目当前定位

ClickDeck 是一个开源 Chrome 插件项目，MVP 目标是：

- 在 Chrome 当前页面上启用可视化编辑覆盖层。
- 用户可以点选页面元素并进行语义化微调。
- 不接入 AI。
- 不回写原始源码。
- 优先导出 PDF，其次导出修改后的 HTML 快照。
- PPT 导出和源码回写作为后续可研方向。

## 2. 总体工程原则

### 2.0 执行代理必须阅读

如果任务交给模型或执行代理，执行前必须阅读：

1. `AGENTS.md`
2. `docs/工程执行原则与协作规则.md`
3. `docs/执行路线图.md`
4. `docs/执行代理操作手册.md`

执行代理必须一次只做一个被明确指定的任务。没有被明确指定任务时，必须停下来询问。

### 2.1 小步推进

每次只完成一个清晰目标，不把多个大功能混在一起做。

推荐节奏：

1. 先写清楚目标和验收标准。
2. 再做最小实现。
3. 跑测试或手动验证。
4. 查看 Git diff。
5. 确认无误后再进入下一步。

### 2.2 先验证风险，再扩功能

优先验证高风险链路：

- Chrome MV3 插件能否稳定注入页面。
- 元素 hover/selection overlay 是否可靠。
- 修改样式是否会破坏页面布局。
- HTML 快照导出是否能独立打开。
- PDF 长页面导出是否可用。

不要一开始就做源码回写、PPT、复杂拖拽或 AI。

### 2.3 保持结构严谨

代码结构要服务于后续维护，避免所有逻辑堆在 content script 里。

建议模块边界：

- `extension/manifest`：插件声明和权限。
- `content`：页面注入、元素选择、DOM/CSS 应用。
- `ui`：浮动工具栏、侧边面板、控件。
- `state`：选中元素、编辑历史、样式 token。
- `export`：HTML/PDF 导出。
- `diagnostics`：日志、错误收集、调试信息。
- `shared`：跨模块类型、工具函数。

实际目录可以在搭项目时按技术栈微调，但边界必须保持清楚。

## 3. Git 协作规则

### 3.1 Git 是每个大步骤的确认点

每完成一个较大的步骤，都要进行一次 Git 检查。

较大的步骤包括：

- 初始化项目脚手架。
- 完成 Chrome 插件基础加载。
- 完成元素选择/描边。
- 完成编辑面板。
- 完成某一类样式修改。
- 完成 HTML 导出。
- 完成 PDF 导出。
- 完成日志/诊断链路。
- 调整重要架构或目录结构。

### 3.2 每步完成后的固定动作

完成一个步骤后执行：

```bash
git status --short
git diff --stat
git diff
```

然后向用户说明：

- 改了哪些文件。
- 每个文件为什么改。
- 当前功能是否通过验证。
- 有没有遗留风险。

用户确认后，再进入下一个较大步骤。

### 3.3 提交策略

正式进入代码阶段后，建议每个稳定步骤形成一个 commit。

commit message 建议格式：

```text
feat: add MV3 extension scaffold
feat: add element selection overlay
feat: add visual style controls
feat: add html snapshot export
feat: add long-page pdf export
docs: define engineering rules
test: add selection overlay tests
```

不把半成品、无关重构和多个大功能混进同一个 commit。

### 3.4 当前仓库状态

当前目录尚未初始化 Git 仓库。正式开始代码前，应先执行：

```bash
git init
git add docs
git commit -m "docs: initialize ClickDeck product docs"
```

如果用户暂时不想创建 commit，也至少要在每个大步骤后查看 diff。

### 3.5 路线图执行台账

`docs/执行路线图.md` 不只是计划文档，也是执行台账。每完成一个较大任务，必须在对应任务章节末尾追加执行记录。

执行记录必须遵守：

- 保留原任务描述，不删除、不覆盖。
- 只在对应章节末尾追加“执行记录”。
- 记录实际完成内容，而不是只写“已完成”。
- 记录验证命令和结果。
- 记录 commit hash。
- 记录遗留风险和下一步。
- 如果实际实现偏离原计划，必须写“订正说明”。

执行记录模板：

```md
### 执行记录 YYYY-MM-DD

- 状态：已完成 / 部分完成 / 阻塞
- 实际改动：
  - ...
- 验证：
  - `npm run build`：通过 / 失败，说明
  - `npm run typecheck`：通过 / 失败，说明
  - `npm test`：通过 / 失败，说明
  - `npm run e2e`：通过 / 失败，说明
- Git：
  - commit: `xxxxxxx message`
- 订正说明：
  - 如无偏离，写“无”。
- 遗留问题：
  - ...
- 下一步：
  - ...
```

如果某一步由模型执行，默认使用两个 commit：先提交任务实现并获取 hash，再把该 hash 写入路线图执行记录并提交文档记录。路线图记录 commit 不得混入代码改动。

## 4. 测试规则

### 4.1 测试分层

项目应尽量建立三层测试：

- 单元测试：验证 style token、DOM 工具函数、导出参数转换。
- 集成测试：验证 content script、状态层、编辑面板之间的数据流。
- 浏览器端 E2E 测试：验证插件加载、元素选择、样式修改、导出入口。

### 4.2 MVP 阶段最低验证

即使早期测试框架还没完整搭好，每个功能也要有最低验证记录：

- 在本地 HTML 页面测试。
- 在 localhost 页面测试。
- 在一个普通网页页面测试。
- 记录成功、失败和异常表现。

### 4.3 推荐工具

后续技术栈确定后再最终决定，但优先考虑：

- TypeScript 类型检查。
- Vitest 做单元测试。
- Playwright 做浏览器行为测试。
- ESLint/Prettier 做基础质量控制。

Chrome 插件 E2E 测试要尽早验证，不要等功能做完才补。

## 5. 日志与诊断规则

### 5.1 日志是一等功能

ClickDeck 会运行在用户页面里，问题可能来自：

- 页面 DOM 结构复杂。
- 网站 CSS 干扰。
- 插件注入失败。
- 权限不足。
- 导出失败。
- 第三方页面安全策略限制。

因此必须从早期就建立诊断链路，而不是等出问题再补日志。

### 5.2 日志分层

建议至少包含以下日志类别：

- `extension`：插件启动、权限、消息通信。
- `selection`：hover、click、当前选中元素。
- `style`：样式 token 变化、应用结果、失败原因。
- `export`：HTML/PDF 导出开始、结束、失败原因。
- `diagnostics`：页面环境、浏览器版本、异常摘要。

### 5.3 日志级别

至少支持：

- `debug`：开发调试细节。
- `info`：关键流程状态。
- `warn`：可恢复异常。
- `error`：功能失败或不可恢复异常。

默认生产状态不刷大量 debug，但开发模式要能打开详细日志。

### 5.4 问题收集包

后续应提供“复制诊断信息”能力，方便用户反馈问题。

诊断信息建议包含：

- ClickDeck 版本。
- 浏览器类型和版本。
- 当前页面 URL 或来源类型（本地、localhost、线上）。
- 当前选中元素摘要。
- 最近 N 条 ClickDeck 日志。
- 最近一次导出参数。
- 最近一次错误堆栈。

注意：诊断信息不应默认上传，也不应默认包含用户页面正文内容。开源项目也要尊重隐私。

## 6. 错误处理规则

所有用户可见失败都要有明确反馈。

不要只在 console 里报错。至少要告诉用户：

- 发生了什么。
- 哪个步骤失败。
- 用户能不能重试。
- 是否可以复制诊断信息。

示例：

```text
PDF 导出失败：当前页面存在跨域资源或打印样式冲突。你可以先尝试导出 HTML，或复制诊断信息提交 issue。
```

## 7. 开源规则

### 7.1 License 倾向

项目默认倾向使用 MIT 或 Apache-2.0。

选择原则：

- 希望最大化二次开发和社区参与：MIT。
- 希望保留更明确的专利授权：Apache-2.0。

在最终确定前，不直接复制 GPL/AGPL 项目的代码。

### 7.2 中英双语原则

ClickDeck 是开源项目，项目说明、使用说明、贡献说明和关键产品文档应尽量同时提供中文和英文。

执行原则：

- 面向用户和贡献者的文档优先中英双语，例如 README、CONTRIBUTING、功能说明、Issue 模板。
- 早期可以先写中文，把产品判断说清楚；当文档稳定后再补英文版本。
- 代码标识、目录名、commit message、测试名优先使用英文，方便国际开源协作。
- 面向开发者的注释默认英文；如果某段产品语境用中文更准确，可以保留简短中文说明。
- 不为了双语而阻塞功能验证，但发布前应补齐核心英文入口。

### 7.3 复用开源项目的规则

可以优先研究：

- VisBug：元素选择、编辑覆盖层、视觉调试交互。
- Stylebot：元素选择、样式应用、样式持久化。
- Onlook：未来源码回写和源码映射方向。
- Paged.js/html2pdf.js：PDF 导出和分页方向。

复用前必须确认：

- License 是否兼容。
- 是否需要保留 attribution。
- 是否会污染项目 License。
- 代码是否能独立拆出，不把无关复杂度带进项目。

## 8. 不做事项

MVP 阶段不做：

- AI 生成内容。
- LLM 接入。
- 源码回写。
- React/Vue/Slidev/Markdown 精准映射。
- Figma 式自由拖拽画布。
- 可编辑 PPT 导出。
- 云端账号系统。
- 默认上传用户页面内容或日志。

## 9. 下一步建议

下一步进入代码前，建议顺序是：

1. 初始化 Git 仓库。
2. 确定 License。
3. 建立 Chrome MV3 + TypeScript + Vite 项目骨架。
4. 做最小插件加载验证。
5. 做元素 hover/click selection PoC。
6. 每完成一步都进行 Git diff 确认。
