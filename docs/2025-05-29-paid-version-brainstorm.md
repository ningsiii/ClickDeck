# ClickDeck 付费版头脑风暴

**日期**：2025-05-29
**参与者**：ningsiii + Claude
**主题**：ClickDeck 付费版的产品方向、核心功能、边界定义

---

## 一、起点：用户反馈与真实痛点

用户反馈：**现在这个产品的位置更改、细节调整更多是细节，但真正的问题是和 AI 沟通时，有时连基本的设计都很糟糕，那些细节显得无关紧要。**

这句话揭示了一个真实的断层。ClickDeck 现在解决的是**生成之后的微调**，但真正的痛苦发生在**生成之前**。AI 生成一个页面，布局乱七八糟、结构不对、信息层级混乱——用户在细节调整（第 10px 的字号修改）之前，已经被第一层的问题击溃了。

现在的 ClickDeck 能力链：
```
用户说一句话 → AI 生成页面 → [巨大的沉默断层] → ClickDeck 微调
```

**> 付费版要填的是那个巨大的断层。**

---

## 二、产品定位：ClickDeck 是什么？

ClickDeck 的核心身份是**翻译官**——在用户和 AI 之间做翻译。

用户的痛点不是"不会调 CSS"，而是**不会跟 AI 沟通自己要什么**。鸡同鸭讲，你不知道谁是鸡谁是鸭，但你知道这就是鸡同鸭讲的事情。

但"翻译"这个词在免费版和付费版里有不同的含义。翻译分两层：

### 第一层：定位翻译（视觉位置 → 代码路径）

把用户在页面上看到的位置翻译成 AI 能理解的 CSS 路径。"这个元素" → `.features > .card:nth-of-type(3)`。

这是基础能力，免费版已经具备。

### 第二层：意图翻译（人的想法 → 结构化指令）

把用户对某个位置的模糊意图翻译成结构化 prompt。"这里加点东西" → "在 `.features > .card:nth-of-type(3)` 之后插入：一段关于产品定价的说明"。

这是付费版的核心。用户不用自己写具体内容，只描述方向，ClickDeck 把位置 + 意图打包给 AI。

**两层合在一起才是完整的翻译。光有第一层，AI 知道位置但不知道要干嘛；光有第二层，AI 知道要干嘛但不知道在哪里干。**

### 免费版：翻译事实（记录器）

用户在页面上动手改了什么东西，系统把改动记录成结构化 prompt。

```
用户：[点击，字号变大，加粗]
系统：fontSize 从 16px 变为 24px，fontWeight 从 400 变为 700
```

AI 知道**发生了什么**。这是从"视觉变化"到"结构化语言"的翻译。

### 付费版：翻译意图（翻译官）

用户指着页面上的某个位置说"这里要加点东西"，系统把意图翻译成结构化 prompt。

```
用户：[选中卡片，点"添加"，写"加一段关于产品定价的说明"]
系统：在 `.features > .card:nth-of-type(3)` 之后插入一段文字。
      意图：添加一段关于产品定价的说明。
      上下文：当前 section 包含 3 张特性卡片，标题为 "Core Features"。
```

AI 知道**要做什么**。这是从"人的意图"到"结构化语言"的翻译。

**> 免费版是记录器——记录你做了什么。付费版是翻译官——翻译你想做什么。从修理工升级为建筑师。**

这个区分类似于人和人之间面对面沟通：我指着电脑的这一行告诉你"你把这一行给我改了"，或者我指着西瓜跟你说"我觉得这玩意太好吃了"。ClickDeck 做的事情就是增加了这个"指"的能力——让 prompt 更加稳定、精确、有锚点。

### 翻译的本质：直译，不是转译

ClickDeck 是**直译**，不是转译。用户说"加一段关于产品定价的说明"，ClickDeck 原封不动地把这句话放进 prompt，再加一个 CSS 路径。ClickDeck 没有加工、没有理解、没有推断。

职责划分：
- 用户负责意图（"我要什么"）
- ClickDeck 负责定位（"在哪里"）+ 打包传递
- AI 负责执行（"具体怎么做"）

**ClickDeck 是邮递员，不是中介。邮递员不为信的内容负责。** 这直接影响用户预期管理——用户知道 ClickDeck 不是在"理解"他的需求，只是在"传递"他的需求。AI 做不好是 AI 的事。

## 三、思考过程：探索过的方向

### 3.1 盖章 / 标注系统（Word 文档批注类比）

**想法**：用户在页面上"盖章"，每个章标记一个位置和意图。类似 Word 文档批注——在某个位置标注"这里要加一段文字"或"这里要放一张图片"。

**进展**：
- 盖章的本质是**在已有元素上标注意图**，定位信息可以复用 ClickDeck 已有的 `ElementLocator` 系统（cssPath、nthOfTypePath、textSnippet 等）
- 应该是"选中一个已有元素 → 盖章"，而不是"点击任意位置"，因为点击任意位置产生的无锚点标注，AI 收到后不知道它指的是什么范围
- 画地为牢——用户选一个区域，在这个区域里写意图

**问题**：
- 如果每一页都要盖章，工作量比现在的"随便改改"更大
- 用户要先**想清楚并写下来**自己要什么——这比动手改更难
- 盖章变成了一种结构化的调研问卷，太重了

**结论**：盖章的方向是对的，但实现方式不应该是"让用户写批注"，而应该是"让用户用点击操作表达意图，系统自动生成结构化信息"。

### 3.2 用户行为轨迹

**想法**：通过记录用户的点击热区、滚动深度、停留时长、鼠标移动路径，作为与 AI 沟通的辅助信号。

**分析**：
- 这是分析工具的能力，不是翻译工具的能力
- 单独做会跟 Hotjar、Microsoft Clarity 竞争（它们是免费的）
- 但可以作为盖章系统的智能建议来源——"你在这个区域停留最久，要不要在这里操作？"

**结论**：不单独做，但可以作为辅助信号。

### 3.3 JSON 格式作为前置信息

**想法**：用 JSON 格式预先告诉 AI 页面的结构和重点，让 AI 生成更好。

**分析**：
- JSON 前置信息变得很重——本质是让用户填写结构化的调研问卷
- 现在的 AI prompt 讨巧，是因为它**后置**——用户先动手改，改完之后系统自动生成结构化信息
- 盖章也应该是这个思路：用户先动手画圈，系统自动生成结构化信息，而不是反过来让用户先填写一堆字段

**> 我之前的 AI prompt 之所以讨巧，是因为它后置——用户先手动改，改完之后系统自动生成结构化信息。盖章也应该是这个思路：用户先动手画圈，系统自动生成结构化信息，而不是反过来让用户先填写一堆字段。**

**结论**：JSON 是盖章系统的**输出格式**，不是输入格式。已经在做了（`change-summary.ts` 里的 `PromptChangeGroup`），不需要额外加。

### 3.4 Prompt 模板库

**想法**：针对不同 AI 工具（Claude、Cursor、v0、Bolt）优化 prompt 格式，提供预设模板（"生成落地页"、"改版博客"等）。

**分析**：
- 有价值但不是核心卖点
- 可以作为付费版的附加功能，但不应该驱动产品定义

**结论**：暂不作为核心卖点，后续根据用户反馈再考虑。

### 3.5 容器拖拽

**想法**：让用户自由拖拽页面元素到任意位置。

**分析**：
- 这是 Figma 级别的功能，意味着：自由定位、碰撞检测、吸附对齐、层级管理、resize handles……是一个完整的布局引擎
- 不是翻译工具应该做的事
- 产品要轻，不要重

**结论**：明确不做。这是别的产品该做的事。

### 3.6 预设标签（标题 / 图片占位 / 卡片 / 按钮）

**想法**：提供预设的元素类型标签让用户选择。

**分析**：
- "标题"、"卡片"、"按钮"这些预设太具象了，反而限制表达
- 更好的预设应该是**动词型的**：添加、删除、替换、移动、缩放

**结论**：用动词型标签替代名词型标签。

## 四、核心方案：动作扩展 + 意图输入

### 4.1 核心原则：描述意图，而非内容

付费版最有价值的交互模式是**"描述意图，而非内容"**。

用户不需要自己写内容（"产品定价：基础版 9.99/月，专业版 19.99/月"），他只需要描述方向（"添加一段关于产品定价的说明"）。ClickDeck 把位置 + 意图 + 上下文打包，AI 来做具体的内容生成和样式判断。

**这个调子可以用在所有操作上**：
- 添加："这里加一段关于产品定价的说明" → AI 来写内容、判断样式
- 删除："这块不要了" → AI 来判断怎么调整布局
- 替换："换成更简洁的版本" → AI 来重写
- 移动："这段挪到上面去" → AI 来处理位置

**> 用户负责意图（人擅长的），系统负责定位（机器擅长的），AI 负责生成（AI 擅长的）。三方各做各擅长的事。**

### 4.2 交互方式：点击为主，文字为辅

基础操作（删除、移动、缩放）**只点击不写字**，跟现在免费版的工作流一致。

"添加"和"替换"操作会弹出一个输入框，用户可以写一句话描述意图。这句话不是必填的——但它是核心价值所在，因为：

**> 自然语言跟 AI 说："在那个卡片下面加一段关于产品定价的说明"→ AI 不知道"那个卡片"是哪个。ClickDeck 说：locator = 精确的 CSS 路径，意图 = "加一段关于产品定价的说明"→ AI 精确知道位置，也知道了内容方向。这就是桥梁的本质。**

打字本身是耗费心神的（否则语音输入法也不会这么流行），但在 ClickDeck 的场景下，用户**只在必要时写一小句**，而不是写一整段需求文档。这跟写一封需求邮件给开发是完全不同的体验。

### 4.3 可执行的动作

| 动作 | 用户操作 | 系统生成的 prompt |
|------|---------|------------------|
| 删除 | 选中元素 → 点"删除" | "删除 locator X 处的元素" |
| 替换 | 选中元素 → 点"替换" → 写意图（可选） | "将 locator X 的内容替换为：意图描述" |
| 在上方/下方添加 | 选中元素 → 点"在下方添加" → 写意图 | "在 locator X 之后插入：意图描述 + 上下文" |
| 移动到 | 选中 A → 点"移到…" → 点 B | "将 locator A 移动到 locator B 的位置" |
| 缩放 | 选中元素 → 点"放大/缩小" | "将 locator X 的尺寸调整为：值" |

### 4.4 空白区域的处理

对于"空白处要加东西"的情况（没有元素可选中），用**相邻元素**来锚定：

- 用户选中最后一张卡片 → 点"在下方添加" → 系统记录"在 `.card:nth-of-type(3)` 之后插入"
- 用户选中页脚 → 点"在上方添加" → 系统记录"在 `footer` 之前插入"

用户还是只点击已有元素，不需要画框。

### 4.5 上下文的价值

系统生成 prompt 时，不仅记录位置和意图，还自动附带**上下文信息**：

```
在 `.features > .card:nth-of-type(3)` 之后插入一段文字。
上下文：当前 section 包含 3 张产品特性卡片，标题为 "Core Features"。
意图：添加一段关于产品定价的说明。
```

AI 看到上下文后，自己就能判断：这段文字应该用什么层级（h3 还是 p）、用什么颜色、放在什么位置。**ClickDeck 不需要做主观判断，它只需要提供足够的上下文让 AI 做出合理的判断。** 这跟"领导模糊给个概念，下属就能完整地做"是一回事——ClickDeck 把模糊概念翻译成了精确的结构化信息。

## 五、产品边界

**做的**：在现有元素上增删改（删除 / 替换 / 在某元素前后添加 / 移动到另一位置 / 缩放）

**不做的**：
- 自由画布拖拽（那是 Figma）
- 全局布局重建（那是重新生成，不是微调工具的活）

**> 对于真正的大改——比如"整个页面的布局从上下结构改成左右结构"——这种操作超出了点击按钮的范畴。诚实的做法是承认：ClickDeck 的边界是"在现有结构上增删改"，不是"重建整个布局"。重建布局是重新生成一轮的事，不应该是微调工具的活。你找错了人，你找错了沟通的对象。你找对了翻译能怎么样？那还是错的。**

## 六、免费版 vs 付费版

| | 免费版 | 付费版 |
|---|---|---|
| **定位** | 记录器——翻译事实 | 翻译官——翻译意图 |
| **核心能力** | 选中元素 → 调样式 → 导出 | 选中元素 → 结构增删改 + 写意图 → 生成方向性 prompt |
| **AI 沟通** | "改这里，fontSize 从 16px 到 24px" | "这里加一段关于产品定价的说明" |
| **翻译层次** | 视觉变化 → 结构化语言 | 人的意图 → 结构化语言 |
| **用户角色** | 修理工 | 建筑师 |
| **交互方式** | 点击 + 按钮调整 | 点击 + 动作按钮 + 可选意图文字 |
| **Prompt 内容** | 样式变更的 before/after | 位置 + 动作 + 意图描述 + 上下文 |

## 七、待验证的问题（已讨论）

1. **Token 花费**：判断——"改来改去改不通"才是最大痛点，不是 token 成本。ClickDeck 的价值是让每一轮沟通更精准。如果一轮说清楚，比十轮说不清楚要便宜得多。
2. **用户沟通不清楚会不会赖产品**：判断——会有一部分。但产品的预期管理应该是"帮你从 10 轮降到 3 轮"，不是"帮你 1 轮搞定"。能减少 50% 的沟通成本已经是巨大价值。
3. **"添加内容"时用户需不需要写字**：判断——这是核心价值所在。用户在必要时写一小句意图描述（"加一段关于产品定价的说明"），不是写具体内容，而是把方向性的活交给 AI。这是自然语言做不到的事（自然语言没有锚点），也是免费版做不到的事（免费版只记录样式变更）。
4. **最小可行版本**：修正——只做"删除"一个动作不够付费的理由。需要至少包含"删除"和"添加"两个操作，其中"添加"的意图输入是付费版的核心卖点。

## 八、下一步

发布最小付费版本：
1. 免费版不变
2. 付费版包含两个核心操作：**删除**和**在某位置添加**
   - 删除：选中 → 点删除 → 自动记录
   - 添加：选中相邻元素 → 点"在下方/上方添加" → 输入意图文字 → 自动记录位置 + 意图 + 上下文
3. 两个操作的记录汇总后生成一份结构化 prompt，格式同现有 AI prompt
4. 发布，看用户反馈
5. 根据反馈迭代（后续扩展：替换、移动、缩放）

**需要的不是更多想法，是一个可以发布的最小版本。**

## 九、让差模型做得更好

ClickDeck 的价值不仅是让好模型更好，也应该让差模型不至于太差。在 prompt 层面能做的：

- **更明确的指令结构**：不是自由文本，而是分字段——位置、动作、意图、上下文，每个字段单独标清
- **约束性语言**：比如"只修改指定元素的样式，不要改变页面其他部分"——减少差模型跑偏的概率
- **示例**：在 prompt 里附带一个 before/after 的范例，让差模型照着格式来

差模型有天花板，但好的 prompt 结构能尽量拉高这个天花板。

## 十、壁垒与工程能力（基于代码分析）

### 当前代码现状

#### ElementLocator 系统（`src/content/dom-utils.ts`）

当前定位器由 `createElementLocator()` 构建，包含两套选择器策略：

- **cssPath**：从目标元素向上查找，遇到第一个带 `id` 的祖先就停下来（`#escapedId`），不够再走 `tag:nth-of-type(N)` 链。**优点**是有 ID 时非常稳定；**缺点**是很多网站用动态生成的 ID（如 `id="r1-a7f3b2"`），不可靠。
- **nthOfTypePath**：完整走 `tag:nth-of-type(N)` 链直到 `<html>`。作为 cssPath 失效时的降级方案。

辅助描述字段：`textSnippet`（前 80 字符）、`imageHint`（仅 `<img>` 的 alt 或文件名）、`classHint`（过滤掉 hash-like 的不稳定 class）、`parentDescriptor`（第一个有意义的祖先：有 id / 稳定 class / 语义化标签）。

#### 上下文提取（`src/content/dom-utils.ts` 的 `getSlideContext`）

当前只做了一件事：向上查找 slide/page 容器（`section`、`.slide`、`[data-slide]` 等），返回 "Slide 4" 或第一个标题文本。**没有提取**兄弟元素信息、section 用途、页面结构概览。

#### Prompt 生成（`src/export/change-summary.ts`）

分组逻辑：按 cssPath 分组，同一元素的多次修改合并。去重逻辑：同一属性多次变更只保留第一个 before 和最后一个 after。回退检测：before === after 的变更会被删除。

#### 没有结构操作

整个代码库只有三种 patch：StylePatch（样式）、ContentPatch（文字内容）、AttributePatch（目前只有图片 src）。**没有删除、添加、移动、克隆元素的任何实现。**

#### 选择器容错

`findElementByLocator` 依次尝试 cssPath 和 nthOfTypePath，都不匹配就放弃返回 null。没有模糊匹配、没有"找最近元素"降级。页面动态内容变化后（如 SPA 路由切换、动态插入列表项），nthOfTypePath 的索引容易失效。

### 当前弱点总结

| 弱点 | 具体表现 | 影响 |
|------|---------|------|
| 定位只到元素级 | textSnippet 只是描述，定位目标永远是整个 DOM 元素；无法定位到 `<p>` 内的某个词 | 付费版"替换"操作无法精确到元素内部 |
| 背景图片不可见 | imageHint 只对 `<img>` 生效；`background-image` 的 div 被当作普通容器 | 用户无法对背景图做操作 |
| 无 Shadow DOM 支持 | parentElement 链无法穿越 Shadow DOM 边界 | 现代组件化页面（React/Vue）中部分元素无法选中 |
| 无 iframe 支持 | 没有 iframe 内的定位逻辑 | 嵌入式内容完全不可用 |
| 上下文太薄 | 只有 slide 标记，没有兄弟元素、section 标题、页面结构 | AI 收到 prompt 后缺乏环境信息来判断样式适配 |
| 选择器脆弱 | 无模糊匹配降级，动态页面下容易定位失败 | 用户修改后重新打开页面，之前的编辑可能无法恢复 |

### 需要加强的工程能力

1. **更精准的定位**
   - 背景图检测：`getComputedStyle(el).backgroundImage` 不为 none 时，记录 url 信息到 locator
   - 细粒度文字定位：记录选中文本在元素内的偏移量和长度（`Range.startOffset` / `endOffset`），而不是替换整个 textContent
   - 层级感知：当一个容器内同时有 `<img>` 和 background-image 时，能区分开

2. **更丰富的上下文提取**
   - 兄弟元素摘要：目标元素的前一个和后一个元素的 tagName + 文本摘要
   - 祖先链语义：不仅找 slide 容器，还要识别 section 的标题、导航栏、页脚等语义区域
   - 页面结构概览：主要 section 列表，让 AI 对页面有全局认知

3. **选择器鲁棒性**
   - 模糊匹配：当 cssPath 和 nthOfTypePath 都失败时，尝试通过 textSnippet + tagName + parentDescriptor 找"最近匹配"
   - 稳定性评分：对选择器的可靠性打分（有稳定 ID > 有语义 class > 纯 nth-of-type），在 prompt 中标注

4. **结构操作的 patch 体系**
   - 新增 `StructurePatch` 类型，支持 delete / insertBefore / insertAfter / moveBefore / moveAfter
   - 结构操作的 undo/redo 需要特殊处理——删除元素要保存完整 DOM 引用以便恢复
   - 结构操作的 prompt 格式需要单独设计（不是"属性从 A 变成 B"，而是"在 X 之后插入 Y"）

这些是工程能力的壁垒，不是一句话能抄走的。但 prompt 本身是明文可见的，不能构成壁垒。当前阶段，ClickDeck 没有强护城河——速度和用户口碑比壁垒更重要。先跑起来，让用户离不开这个工作流，再慢慢建立壁垒。

## 十一、产品专家视角：待解决问题

### 11.1 免费版太窄

**问题**：免费版只改样式（字号、颜色、间距）。但用户真正的痛点——"跟 AI 沟通不清楚"——在免费版里完全没有被触及。用户下载后不知道付费版能做什么。

**产品态度**：免费版不是诱饵，它本身就应该是一个好产品。用户用了免费版，觉得"这东西有用，不错"，然后发现付费版能做更多事情。不是一个残缺品在逼你升级，而是一个完整的东西有更高级的版本。

**讨论后的思路**：
- 免费版解决一类问题（样式微调），付费版解决另一类问题（结构增删 + 意图表达）。两层翻译各自独立成立，付费版是能力扩展，不是功能解锁
- 免费版导出样式变更 prompt → 本身对用户就有用，比自然语言描述更精确
- 付费版加了第二层翻译（意图）→ 不是"解锁"，是"你能做更多事了"
- 免费版 = 第一层翻译（定位 + 样式记录）；付费版 = 第一层 + 第二层（定位 + 意图）

**甜头方案（定位展示 + 可重复操作）**：

免费版给用户展示定位能力：选中一个元素 → ClickDeck 展示它的位置信息（在哪个 section、左边是什么、右边是什么）→ 免费版用户觉得"有用"。

付费版的操作是**可重复的、跟字号按钮平级的**。不是"进入第二层的入口"，而是跟样式微调一样的日常操作流：
- 用户选中第一页的某个元素 → 点"在下方添加" → 写意图 → 记录
- 选中第二页的另一个元素 → 点"删除" → 记录
- 选中第三页的元素 → 点"替换" → 写意图 → 记录
- 最后所有操作汇总成一份 prompt

每次操作都是独立的、可重复的、可以在不同页面不同元素上做的。跟免费版的工作流完全一致——免费版是选中元素 → 点字号增大 → 选中另一个 → 点颜色改变。付费版是选中元素 → 点"添加" → 写意图 → 选中另一个 → 点"删除"。

操作时展示的位置信息本身就是意图输入框的引导——用户看到了上下文，自然知道自己想干嘛。不需要额外的引导文案。

**待验证**：免费版当前的样式微调能力是否已经足够让用户觉得"有用"？还是需要扩展免费版的能力范围？

### 11.2 付费版的定价锚点

**问题**：用户凭什么为"删除+添加"付费？

**讨论后的思路**：
- 定价锚点是**意图识别**——ClickDeck 做的是两层翻译，免费版只做到第一层（定位），付费版加上第二层（意图）
- 定位本身不值钱（免费版已有），值钱的是"把人的模糊意图翻译成结构化指令"
- 定价故事："你用自然语言跟 AI 说'那个卡片下面加点东西'，AI 不知道'那个'是哪个，也不知道'东西'是什么。ClickDeck 帮你精确到 CSS 路径 + 你的意图描述"
- ningsiii 已经加了一些用户，可以做用户访谈验证用户到底为哪个具体场景买单

### 11.3 竞争风险

**问题**：如果 Cursor、v0、Bolt 自己做了"点击元素 → 生成 prompt"怎么办？

**讨论后的思路**：
- ClickDeck 跨平台——今天用 v0，明天换 Cursor，ClickDeck 都能用。单个 AI 工具不愿意建跨平台能力
- Cursor/v0 做 Chrome 插件 = 从"AI 编码工具"变成"浏览器插件公司"，不是它们的核心业务
- ClickDeck 天然配 Chrome，生成的是 HTML——这是网页的本体，HTML 本身就是跨平台的，不需要额外防御
- 但诚实地说，壁垒是薄的。速度和用户口碑比技术壁垒更重要

### 11.4 最小版本范围

**问题**：最小版本应该包含多少功能？

**讨论后的思路**：
- 只做"删除"不够——锚点更不清，用户不会为单一删除操作付费
- 需要至少包含"删除+添加"两个操作：删除证明能做结构操作，添加证明意图输入有价值，两个缺一不可
- 文档第八节的方案（删除+在某位置添加）保持不变
- 这已经是最小版本，不能再小——再小就只有定位没有意图，等于免费版

### 11.5 ⚠️ 核心未解问题：免费版和付费版的边界理由

**问题**：为什么免费版只能做第一层（定位 + 样式记录），付费版能做第二层（意图识别）？技术上没有硬性壁垒——免费版也可以加一个输入框让用户写意图。翻译插件的付费理由很清楚（API 成本、更好的模型），但 ClickDeck 的边界是人为划定的。

**尝试过的区分**：
- 免费版的操作结果在 ClickDeck 里可见（改完字号你看得到），付费版的操作结果需要交给 AI 之后才可见（写意图后 AI 生成内容）——但这不够硬
- 免费版是"改已有内容"，付费版是"描述还没有的东西"——也是一个角度，但不是不可逾越的理由

**状态**：这个问题没有想通。可能解决了这个问题之后，前面的一些判断会变得更通顺。待后续灵感。

### 11.6 探索方向：Skill 机制

**想法**：ClickDeck 加入"skill"机制——不是让 AI 自主做决定，而是一套规则手册，增强 prompt 的质量。

**两种方向**：
1. **Skill 给插件**：规则增强 ClickDeck 生成 prompt 的方式。比如"在 hero section 添加内容时，自动附带移动端适配提示"。ClickDeck 按规则打包更完整的指令给 AI。
2. **Skill 给大模型**：ClickDeck 不只生成 prompt，还生成一份针对当前页面的"规则集"——"这个页面用卡片布局，3 个 section，修改时保持 grid 结构"。用户可以把这份规则集复用在同项目的不同页面上。

**个人风格 Skill（远期方向）**：

用户说不出来自己的风格，但他的操作记录了他的风格。他每次都把字号调到 24px，每次都把间距压得很紧，每次都用某种配色——这些操作就是他的审美指纹。ClickDeck 不需要他写 prompt 描述自己的风格，ClickDeck 从他的行为里自动提取。

这正好是 ClickDeck 最擅长的模式——后置。用户先动手改，系统自动生成结构化信息。skill 也一样：用户先动手操作，系统自动生成他的个人风格规则。

通用模板谁都能用，不值钱。个人风格是从你自己的操作里提炼出来的，别人拿不走。用得越久，ClickDeck 越懂你，迁移成本越高——这可能是真正的护城河。

**风险**：可控 vs 不可控。ClickDeck 现在的 prompt 是完全确定性的——同样的操作永远生成同样的 prompt。加了规则后，输出取决于"哪条规则被触发"，越复杂越不可预测。

**与 11.5 的关联**：Skill 可能是解决免费版/付费版边界的一个方向——免费版生成标准 prompt，付费版生成带规则增强的 prompt。个人风格 Skill 也可能是独立的付费理由（你的风格，从你的操作中来，别人拿不走）。但不确定，需要进一步想。

**工程预埋方向**：

现有 patch 数据（属性名、before/after 值、目标元素描述）已经包含了量化的个人风格信息。用户每次把字号从 16 改到 24，改了足够多次之后，"偏好 24px"就是他的风格。

预埋要点：
- Patch 存储要跨页面、跨项目汇总，不只是当前页面
- Patch 加 section 标注（hero、card grid、footer 等），区分不同区域的偏好
- 现有 prompt JSON 格式不需要改，分析是离线做的

真正的分析引擎和规则生成是后面的事。预埋成本不大。

**状态**：远期方向。需要用户量、数据积累、个人数据存储。优先解决 11.5 的核心边界问题和最小版本发布。

## 十二、2026-06-01 收敛结论：从付费版发散到可控修改中转层

### 12.1 结论

0529 的多数内容属于付费版发散，但其中“定位 + 意图 + 上下文”的方向已经被后续真实用户反馈验证。它不应再被理解为单纯的付费版功能，而应收敛为 ClickDeck 下一阶段的父级方向：

**ClickDeck 是 AI HTML 的可控修改中转层。**

它不是 Figma，不是 Canva，不是全自动 AI 设计工具，也不是直接替用户重写页面的生成器。它要做的是：

- 把用户指向的页面位置表达准确。
- 把用户想做的事情表达清楚。
- 把当前页面、当前区域、相邻元素、当前样式等上下文补齐。
- 把约束条件写清楚，降低 AI 改坏整页的概率。
- 让用户能先完成确定性的基础样式修改，再把抽象修改交给 AI。

### 12.2 与 0531 用户调研的关系

0531 用户调研解决的是“改完以后如何展示和交付”，对应 HTML 交付、长图、演示模式、图片版 PDF。

0529 中被保留下来的主线解决的是“AI 生成的半成品 HTML 如何被可控地改到值得交付”。

两者不是互相替代，而是前后衔接：

```text
AI 生成 HTML
↓
ClickDeck 可控修改
↓
ClickDeck 展示与交付
```

任务 44A 至任务 48 已经完成“展示与交付”的第一轮闭环。下一阶段应回到“可控修改”本身，但不能把边界扩张成自由画布或复杂排版引擎。

### 12.3 产品分层

#### 第一层：确定性样式编辑

用户能直接判断结果，ClickDeck 可以立即修改 DOM。

现有能力已经覆盖字号、字重、行高、字间距、对齐、颜色、背景、圆角、间距、图片尺寸和图片裁切等基础项。后续应重点研究：

- 字体族：中文黑体、宋/雅宋风格、楷体风格，英文 Sans、Serif、Mono。
- 字号控制：更明确的上下限、大小步进、常用字号预设。
- 透明度：元素透明度、图片透明度、容器透明度。
- 阴影：无、轻、卡片、浮层等预设。
- 边框：宽度、颜色、样式。
- 背景：自定义背景色优先，渐变暂缓。

字体需要单独说明限制：浏览器只能使用页面已加载字体、本机已安装字体或 CSS fallback 栈。ClickDeck 可以提供字体族 fallback 预设，但不能保证不同电脑上显示完全一致，除非后续单独实现字体资源内联或云字体注入。

这一层必须克制。ClickDeck 可以参考 PPT 的信息层级：字体、字号、颜色这类高频项放在一层；阴影、透明度、边框细节放在二层。但 ClickDeck 不应照抄 PPT、PDF 编辑器或完整设计软件的能力集合。更合适的克制程度应接近轻量插件：能快速解决半成品 HTML 的高频小改，而不是把面板做成 Photoshop 工具箱。

判断标准：

- 高频：AI 生成 HTML 后经常需要微调。
- 可见：用户点完能立即判断效果是否更好。
- 可撤销：必须进入现有 undo/redo 和 patch 记录。
- 低风险：不破坏整页结构，不引入布局引擎。
- 轻心智：需要解释 30 秒以上的能力先不做。

面板分层建议：

- 一层面板：Default、Sans、Serif、CJK Serif、Mono 等跨语言字体风格预设；A-/A+ 与少量字号预设；B-/B+；颜色、背景色、对齐、基础间距、圆角。
- 二层或高级折叠项：透明度、阴影预设、边框预设、背景色自定义、margin/padding 细控制、自定义字体名。
- 暂缓：渐变编辑器、完整字体管理、任意 CSS 输入、复杂四方向间距面板、PPT 式对齐分布、图层、组合、吸附、自由拖拽排版。

国际化和字体预设也不能只按中文视角设计。Chrome 插件不会自动生成多语言 UI 文案，正式上架需要使用 `_locales` 管理文案。字体预设应按跨语言风格命名，不应只写“雅宋”“文楷”等具体中文字体；CJK 字体可以放在 fallback 栈里，但不能承诺每台机器显示完全一致。

#### 第二层：上下文增强

用户表达意图之前，ClickDeck 要先把“这里”讲清楚。

后续 prompt 不应只有 selector 和 before/after，而应包含：

- 目标元素描述。
- 当前元素文本或图片信息。
- 当前元素关键 computed style。
- 所在 slide 或 section。
- 前后兄弟元素摘要。
- 父级语义区域。
- 页面结构概览。
- 选择器稳定性提示。
- 限制 AI 修改范围的约束。

这一层是 ClickDeck 的核心壁垒之一。prompt 本身容易被复制，但稳定定位、上下文抽取、约束组织、用户操作记录不容易一次性抄走。

#### 第三层：批注式意图快递

用户不应该被要求写复杂需求文档，也不应该进入自由拖拽画布。

推荐交互是：

```text
选中已有元素
↓
选择动作
↓
输入一句意图
↓
ClickDeck 生成结构化 prompt
```

第一批动作可以是：

- 在上方添加。
- 在下方添加。
- 替换这里。
- 删除这里。
- 调整风格。
- 移动到某个参照元素附近。

空白位置不通过自由框选解决，而是通过相邻元素锚定，例如“在这个卡片下方添加”或“在这个标题上方添加”。

### 12.4 快递员/翻译官边界

ClickDeck 只对“表达是否准确”负责，不对“AI 最终审美是否完美”负责。

应在产品和 prompt 中反复强调：

- 用户负责意图。
- ClickDeck 负责定位、上下文、约束、打包。
- AI 负责生成或修改。

ClickDeck 的承诺不是“帮你一轮改好”，而是“让你从自然语言里说不清楚的修改，变成 AI 更容易执行的结构化修改请求”。

这也解释了为什么不能直接做自由拖拽：

- 自由拖拽意味着布局引擎、碰撞处理、响应式适配、层级管理、吸附对齐。
- 这会把 ClickDeck 推向 Figma/Canva 的方向。
- ClickDeck 更合理的做法是记录“把 A 移到 B 附近”这种意图，并附带 A/B 的定位和上下文，让 AI 或上游代码工具处理具体实现。

### 12.5 下一步不应直接实现付费版

当前不建议直接把本方向包装为“付费版”并开始做收费边界。原因：

- 付费边界仍未完全想清楚。
- 用户真实需求先指向“可控修改能力”，不是先指向“订阅体系”。
- 过早收费容易让免费版变成残缺品，伤害增长。

更合理的下一步是先做三个可研：

1. 样式编辑增强可研。
2. 上下文增强可研。
3. 批注式意图交互可研。

可研完成后，再拆成具体 MVP 执行任务。低等级模型不得跳过可研直接实现结构编辑或意图编辑。

### 12.6 暂缓方向

- 暂缓自由画布拖拽。
- 暂缓复杂容器排版引擎。
- 暂缓全局自动美化。
- 暂缓 PPTX 导出。
- 暂缓复杂付费体系。
- 暂缓个人风格 Skill 的实现，但保留 patch 数据与上下文数据的工程预埋价值。
