# ClickDeck 下一阶段目标

## 背景

近期通过用户反馈发现，用户已经不只是使用 ClickDeck 修改 HTML。

用户开始将 HTML 用于：

* 给自己阅读
* 给同事查看
* 给领导汇报
* 导出长图
* 导出 PDF
* 现场展示

说明 ClickDeck 正在从 HTML 编辑工具逐步变成 HTML 工作流工具。

工作流：

AI生成HTML
↓
ClickDeck编辑HTML
↓
展示HTML
↓
交付HTML

---

## 调研发现

### 发现1

部分用户直接使用 HTML。

场景：

* 自己阅读
* 同事查看

说明 HTML 本身已经是一种交付格式。

---

### 发现2

用户存在导出需求。

原因：

* 手机兼容性问题
* 领导无法直接打开 HTML
* 需要归档和转发

说明需要提供 HTML 之外的交付形式。

---

### 发现3

用户更关注展示效果一致性。

用户反馈：

* 希望导出效果尽量接近 HTML
* 希望长卷连续阅读
* 不喜欢自动分页
* 不希望出现空白页

说明用户关注的是高保真展示。

---

### 发现4

目前没有明确证据表明用户需要可编辑 PDF。

用户更关注：

* 查看
* 阅读
* 汇报
* 分享

而不是：

* 编辑 PDF

---

## 本阶段尝试

### 方向1：HTML编辑

目标：

继续完善 HTML 编辑体验。

---

### 方向2：长图导出

目标：

支持高质量长图导出。

场景：

* 微信发送
* 手机查看
* 快速分享

---

### 方向3：PDF导出

目标：

提供稳定的交付版本。

优先级：

高保真展示
＞可编辑能力

优先尝试：

图片版 PDF

而非可编辑 PDF。

---

### 方向4：展示模式

目标：

让 HTML 可以直接用于汇报。

场景：

* PPT替代
* 现场展示
* 分享演示

---

## 2026-05-31 补充结论：四条交付路径

这次调研后的核心判断是：ClickDeck 不应该继续把“PDF 打印”当作唯一交付出口。用户真实需要的是把已经改好的 HTML 交付出去，并根据场景选择不同形态。

后续产品路径拆成四条，分别处理，不混在一个按钮或一个任务里。

### 路径 1：HTML 交付

定义：

用户在当前网页里用 ClickDeck 修改内容和样式后，导出一个可以发送给同事或领导在电脑上打开的 HTML 快照。

适用场景：

* 自己保存一版改好的页面
* 发给同事在电脑上查看
* 发给领导在电脑上预览
* 保留 HTML 原本的动画、交互和视觉效果

必须说清楚：

* 这是导出当前页面快照，不是把修改写回原始源代码。
* 这是 HTML 交付，不是 AI 改代码，也不是让 AI 重新生成页面。
* ClickDeck 已经落到 DOM 或 inline style 上的改动，应尽量保留在导出的 HTML 中。
* 如果页面依赖外部字体、外部图片、远程脚本或远程 CSS，导出 HTML 不保证完全离线可用，除非后续任务专门做资源内联。

第一阶段验收标准：

* 修改文字字号、颜色、圆角、图片后导出 HTML，再打开导出的 HTML，能看到这些修改。
* 导出的 HTML 不包含 ClickDeck 面板、选中框、调试 UI。
* 导出入口文案必须写清楚是“导出 HTML”，不能只写“导出”。
* 如果无法保证离线资源完整，界面或文档中必须明确说明限制。

### 路径 2：长图导出

定义：

把当前页面导出为一张连续、干净、高清的长图，用于分享和快速查看。

适用场景：

* 微信发送
* 手机查看
* 快速分享给不方便打开 HTML 的人
* 把长页面或报告作为图片交付

必须说清楚：

* 长图导出只做“一张连续长图”。
* 长图导出不做 A4。
* 长图导出不做 16:9。
* 长图导出不做自动分页。

第一阶段验收标准：

* 导出的图片是连续的，不出现额外空白页。
* ClickDeck 面板、选中框、调试 UI 不出现在图片中。
* 图片清晰度足够用于普通屏幕阅读。
* 页面滚动位置在导出结束后恢复。

### 路径 3：演示模式

定义：

针对 PPT-like HTML 提供浏览器内演示能力，让用户不必把 HTML 转成 PPT，也能像放映 PPT 一样展示页面。

适用场景：

* HTML 本身就是一页一页的 slide deck。
* 用户需要现场汇报。
* 用户希望保留 HTML 动画、动效、交互和视觉质感。

必须说清楚：

* 演示模式只面向 PPT-like HTML。
* 演示模式不面向普通长页面。
* 演示模式不是 PPTX 导出。
* 演示模式不是自由画布。

第一阶段验收标准：

* 能进入全屏或接近全屏的展示状态。
* 能用键盘左右键、上下键或空格切换页面。
* 能用 Esc 退出。
* 进入演示时隐藏 ClickDeck 面板和编辑辅助 UI。
* 不破坏原页面动画和交互。
* 对普通长页面应给出清楚提示或不显示入口，不要强行把长页面当 PPT。

### 路径 4：图片版 PDF

定义：

PDF 只作为固定化保存和归档格式，不追求可编辑。优先考虑用截图或图片切片生成视觉稳定的 PDF。

适用场景：

* 归档
* 发给必须接收 PDF 的对象
* 保存一份固定视觉结果

必须说清楚：

* 图片版 PDF 不是可编辑 PDF。
* 图片版 PDF 不保证文字可选中。
* 图片版 PDF 的目标是视觉接近 HTML，而不是保留 DOM 或文本结构。
* 旧的浏览器打印引擎路线暂时不再作为主线继续深挖。

后续应拆成三种模式验证：

* 长页 PDF：适合长页面阅读和归档。
* A4 图片 PDF：适合传统文档归档。
* 16:9 图片 PDF：适合 slide deck 固定化保存。

第一阶段验收标准：

* 先完成可研，明确是否复用长图截图链路。
* 确认图片尺寸、PDF 页数、文件体积和清晰度的取舍。
* 确认是否需要新增依赖；如果需要，必须单独说明理由。

## 2026-05-31 优先级

当前优先级从高到低：

1. 旧 PDF 打印入口降级：先避免普通用户继续误用已经暂停的旧打印链路。
2. 演示模式 MVP：因为它直接补上 HTML 无法变成 PPT 但又需要汇报的缺口。
3. 长图导出技术预研：因为长图截图链路也会成为图片版 PDF 的技术铺垫。
4. 长图导出 MVP：因为它解决分享、手机查看和快速转发。
5. HTML 交付稳定化：因为 HTML 本身已经是核心交付格式，需要把“导出 HTML”的含义和稳定性讲清楚。
6. 图片版 PDF 可研：先确认技术路线，不再继续靠浏览器打印引擎硬补。
7. 图片版 PDF MVP：在可研完成后再执行，避免再次陷入不稳定的打印补丁。

暂时不要继续投入的方向：

* 不做可编辑 PDF。
* 不做 PPTX 导出。
* 不做把修改自动写回源代码。
* 不做高级 AI 编辑能力。
* 不继续把浏览器打印引擎作为 PDF 主线深挖，除非后续用户明确要求恢复。

旧 PDF 打印代码处理原则：

* 不立刻删除旧 PDF 代码和 debug 链路，避免大范围回归。
* 普通用户面板里应隐藏旧 PDF 打印按钮。
* 如果需要保留，只能作为明确的实验/调试入口。
* 新的 PDF 方向应转向图片版 PDF，而不是继续修补旧 `window.print()` 路线。
