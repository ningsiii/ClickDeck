# ClickDeck 执行代理操作手册

这份文档是给后续执行模型/执行代理看的。目标是降低误改、乱改、跳步骤、漏验证、漏记录的概率。

## 1. 执行前必须阅读

每次开始执行前，必须按顺序阅读：

1. `docs/工程执行原则与协作规则.md`
2. `docs/执行路线图.md`
3. `docs/执行代理操作手册.md`
4. 当前被指定任务对应的章节
5. 与任务直接相关的源码文件

如果没有被明确指定任务，只能停下来询问，不允许自己选择一个任务开做。

## 2. 当前项目边界

ClickDeck 当前是：

- Chrome/Edge 扩展。
- 在当前页面上做可视化微调。
- MVP 不接入 AI。
- MVP 不回写源码。
- MVP 不做可编辑 PPT。
- MVP 不做自由拖拽画布。

执行代理不得把这些非目标偷偷加回来。

## 3. 单任务锁定规则

一次只允许执行一个任务。

例如用户说“执行任务 1”，就只能做：

- 拆分 `src/content/main.ts`
- 保持行为不变
- 更新路线图任务 1 的执行记录
- 验证
- commit

不得顺手做：

- 新 UI 设计
- HTML/PDF 导出
- undo/redo
- 日志缓冲
- 大规模重构
- 依赖升级
- 格式化整个项目

如果执行过程中发现另一个问题，只记录到“遗留问题”，不要顺手修，除非它阻塞当前任务验证。

## 4. 文件修改边界

执行任务前先判断允许修改哪些文件。

### 任务 1 推荐允许修改

- `src/content/main.ts`
- `src/content/controller.ts`
- `src/content/overlay.ts`
- `src/content/panel.ts`
- `src/content/selection.ts`
- `src/content/style-actions.ts`
- `src/content/dom-utils.ts`
- `src/shared/messages.ts`
- `src/shared/types.ts`
- `docs/执行路线图.md`

任务 1 默认不应修改：

- `public/manifest.json`
- `vite.config.ts`
- `package.json`
- `package-lock.json`
- `README.md`
- `docs/PRD文档`
- `docs/Chrome插件：HTML展示轻量级编辑.md`

如果确实必须改默认不应修改的文件，必须在执行记录里写清楚原因。

## 5. 禁止事项

执行代理不得：

- 删除原有文档内容。
- 覆盖路线图原计划。
- 跳过测试。
- 测试失败还提交。
- 只看 `git diff --stat` 不看具体 diff。
- 把多个任务混进一个 commit。
- 大面积格式化无关文件。
- 新增未经请求的大依赖。
- 改 License。
- 改产品方向。
- 删除 E2E。
- 降低 TypeScript strictness。
- 为了让测试通过而删测试。
- 在没有说明的情况下修改 `.gitignore`、构建配置或 npm scripts。

## 6. 允许的小补丁

只有以下小补丁可以在当前任务内顺手做：

- 修复当前任务引入的类型错误。
- 修复当前任务引入的测试失败。
- 调整当前任务新增模块的导入路径。
- 为当前任务新增必要的最小单元测试。
- 补当前任务对应的路线图执行记录。

其他发现都放到“遗留问题”。

## 7. 执行流程

每个任务必须按这个顺序执行：

1. 读取规则和路线图。
2. 确认 Git 工作区是否干净：

```bash
git status --short
```

3. 如果工作区不干净，先停下来报告，不要覆盖别人的改动。
4. 只修改当前任务需要的文件。
5. 运行验证：

```bash
npm run build
npm run typecheck
npm test
npm run e2e
```

6. 如果验证失败，修复当前任务相关问题后重跑。
7. 查看 diff：

```bash
git status --short
git diff --stat
git diff
```

8. 暂存本任务代码和测试文件，不包含路线图执行记录。
9. commit 本任务实现。
10. 读取刚创建的 commit hash。
11. 在 `docs/执行路线图.md` 对应任务章节末尾追加执行记录，记录第 9 步的实现 commit hash。
12. 单独 commit 路线图执行记录。
13. 再次确认工作区干净：

```bash
git status --short
```

14. 汇报结果，不进入下一任务。

## 8. 完成定义

一个任务只有同时满足以下条件，才算完成：

- 完成了任务章节中的目标。
- 没有加入任务外功能。
- `npm run build` 通过。
- `npm run typecheck` 通过。
- `npm test` 通过。
- `npm run e2e` 通过，或清楚说明为什么本任务无法跑。
- `docs/执行路线图.md` 已追加执行记录。
- 已创建任务实现 commit。
- 已创建路线图执行记录 commit。
- 工作区干净。

少一项都不算完成。

## 9. 阻塞处理

遇到以下情况必须停止并报告：

- Git 工作区一开始就不干净。
- 不知道当前任务是哪一个。
- 需要改任务边界外的大量文件。
- 需要新增大依赖。
- E2E 因环境问题无法运行。
- 修复当前问题会改变产品方向。
- 需要删除或弱化测试才能通过。

报告格式：

```text
阻塞原因：
已尝试：
当前状态：
建议下一步：
```

## 10. 执行记录要求

每个任务完成后，在 `docs/执行路线图.md` 对应任务章节末尾追加：

```md
### 执行记录 YYYY-MM-DD

- 状态：已完成 / 部分完成 / 阻塞
- 实际改动：
  - ...
- 验证：
  - `npm run build`：通过 / 失败，说明
  - `npm run typecheck`：通过 / 失败，说明
  - `npm test`：通过 / 失败，说明
  - `npm run e2e`：通过 / 失败，说明
- Git：
  - commit: `xxxxxxx message`
- 订正说明：
  - 如无偏离，写“无”。
- 遗留问题：
  - ...
- 下一步：
  - ...
```

执行记录必须写在对应任务章节里，不要写到文档最底部。

## 11. Commit 规则

为了让路线图执行记录能写入准确 commit hash，每个任务默认使用两个 commit：

1. 任务实现 commit：包含代码、测试、必要 fixture。
2. 路线图记录 commit：只更新 `docs/执行路线图.md` 的对应执行记录。

示例：

```text
refactor: split content script modules
docs: record task 1 execution
```

禁止把多个任务合进一个 commit。

禁止在路线图记录 commit 里混入代码改动。

## 12. 任务 1 专用提示词

如果要把任务 1 交给执行代理，可以直接使用：

```text
请执行 ClickDeck 的任务 1：拆分 content script。

执行前必须阅读：
- docs/工程执行原则与协作规则.md
- docs/执行路线图.md
- docs/执行代理操作手册.md

任务范围：
- 只拆分 src/content/main.ts。
- 保持现有行为不变。
- 不新增导出。
- 不新增 undo/redo。
- 不调整产品视觉。
- 不修改 manifest、package、vite 配置，除非验证被阻塞且必须说明原因。

建议拆分为：
- src/content/controller.ts
- src/content/overlay.ts
- src/content/panel.ts
- src/content/selection.ts
- src/content/style-actions.ts
- src/content/dom-utils.ts
- src/shared/messages.ts
- src/shared/types.ts

验收：
- Alt+Shift+C 仍能启用/停用。
- Edge E2E 仍通过。
- main.ts 明显变薄。
- npm run build 通过。
- npm run typecheck 通过。
- npm test 通过。
- npm run e2e 通过。

完成后：
- 在 docs/执行路线图.md 的任务 1 章节末尾追加执行记录。
- 先提交实现，建议 message: refactor: split content script modules
- 再提交路线图记录，建议 message: docs: record task 1 execution
- 最后报告 git diff --stat、验证结果和 commit hash。
- 不要进入任务 2。
```
